# CLI 框架

## 1. 问题

用户需要通过命令行运行 decay 来获取项目健康报告。项目初始化完成后，虽然代码可编译，但还没有命令行入口——用户无法通过 `decay` 命令与工具交互。需要建立 CLI 框架作为所有用户交互的入口。

## 2. 目标用户

使用 Claude Code 的开发者和 AI agent，在终端中运行 `decay` 命令。当前没有可执行的 CLI 入口，用户无法调用工具。

## 3. 核心假设

**提供 `decay --help` 入口 → 用户能发现工具的使用方式，后续功能（采集、评分、处方）可通过统一的 CLI 框架暴露给用户。**

验证方式：运行 `decay --help` 输出帮助信息，退出码 0。

## 4. 方案

- **Before**: 项目可编译但无命令行入口，用户无法运行 decay → **After**: 运行 `decay --help` 输出帮助信息，为后续功能预留命令扩展点

## 5. 验收标准

- 运行 `decay --help` → 输出包含工具名称和用法说明，退出码 0
- 运行 `decay --version` → 输出版本号
- 运行 `decay`（无参数）→ 不 panic，输出帮助或提示信息

## 6. 排除项

- 不包含子命令实现（采集、评分等功能在后续里程碑）
- 不包含 `--json` 输出格式（→ M6）
- 不包含 `--debug` flag（→ M7）
