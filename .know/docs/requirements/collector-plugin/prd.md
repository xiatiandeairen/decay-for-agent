# 采集层插件化

## 1. 问题

现有采集层由两个独立的 `collect()` 函数组成（`scan.rs` 和 `git.rs`），各自直接写入固定的 DB 表。v3 新增 5 个维度需要新的数据源：文件内容模式匹配、依赖文件解析等。当前架构每加一个数据源需要：新建采集函数 + 新建 DB 表 + 在 `run.rs` 中手动调用，扩展成本高。

## 2. 目标用户

decay 开发者。采集层插件化后，新增数据源只需实现 Collector trait + 注册，无需修改调度代码。

## 3. 核心假设

**将采集器抽象为统一 trait + 注册表 → 新增数据源的实现成本从"改 3+ 文件"降为"新增 1 个文件 + 注册 1 行"。**

验证方式：重构后现有两个采集器（file scan、git analysis）通过 trait 注册表统一调度，输出与重构前一致。

## 4. 方案

- **Before**: scan.rs 和 git.rs 是独立函数，各自硬编码 DB 表，run.rs 手动调用并处理错误
- **After**: 每个采集器实现 Collector trait，通过注册表统一调度，DB 表由采集器自行管理

## 5. 验收标准

- `cargo test` 全部通过
- `decay` 输出与重构前一致
- 新增采集器只需：创建文件、实现 Collector trait、在注册表中注册
- 现有 scan.rs 和 git.rs 的内部实现不变（仅包装为 trait 实现）
- git 采集失败时其他采集器不受影响（错误隔离）

## 6. 排除项

- 不新增任何采集器（v3 M4-M8 各自负责）
- 不改 Dimension trait（M1 已完成）
- 不改 DB schema 基础结构（各采集器自行建表）
- 运行时动态加载（编译期注册已满足需求）
