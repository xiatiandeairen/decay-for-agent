# complexity 评分

## 1. 问题

项目复杂度的核心信号是大文件集中和深层嵌套——大文件意味着职责过重，深嵌套意味着模块耦合。M2 采集了每个文件的大小和深度，但需要将这些指标转化为一个可量化的复杂度分数。

## 2. 目标用户

使用 decay 的 AI agent。agent 运行 `decay` 后获得 complexity 分数，用于判断是否需要拆分大文件或简化模块结构。

## 3. 核心假设

**基于文件大小分布和嵌套层级计算 complexity 分数 → 用户能识别复杂度热点，在文件膨胀失控前介入。**

验证方式：运行 `decay`，输出包含 complexity: XX/100 分数。

## 4. 方案

- **Before**: 知道哪些文件大但不知道整体复杂度水平 → **After**: 输出 complexity 0-100 分，量化复杂度健康程度

## 5. 验收标准

- 运行 `decay` → 输出包含 complexity 分数（0-100）
- 大文件占比越高、平均文件越大、深层文件越多 → 分数越低
- 对文件均匀分布的项目 → 分数 >80

## 6. 排除项

- 不包含 AST 级复杂度分析（圈复杂度等 → 语言增强，后续版本）
- 不包含目录结构评分（→ structural 维度）
- 不包含问题诊断和重构建议（→ M4）
