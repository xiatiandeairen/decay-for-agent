# 数据采集

## 1. 问题

### 痛点

项目衰退信号隐藏在文件结构和 git 历史中，人工无法系统性提取。同时采集层架构缺乏统一抽象，每新增一个数据源需要改多个文件，扩展成本高。

- **文件结构盲区**: 无自动化手段采集文件膨胀和目录层级失控等结构指标
- **变更模式不可见**: 频繁修改的热点文件、高 churn 率模块隐藏在 git log 中
- **采集层扩展成本高**: scan.rs 和 git.rs 独立函数硬编码 DB 表，每加一个数据源需改 3+ 文件

### 影响范围

所有项目都有文件结构和 git 历史，每次 decay 运行都需要扫描。structural 维度依赖文件扫描数据，fragility 维度依赖 git 分析数据。v3 新增 5 个维度需要新数据源，采集层扩展问题更突出。

### 为什么现在做

文件结构和 git 历史数据是评分维度的数据基础。不做则评分无数据来源。采集层不统一则每个新数据源都要改 3+ 文件。

## 2. 目标用户

| 角色 | 场景 | Before | After |
|------|------|--------|-------|
| AI agent | 运行 `decay` 后自动获得项目文件结构量化数据 | 不知道项目有多少文件、目录多深、哪些文件过大 | 每次运行自动扫描文件树，结构指标写入快照 |
| AI agent | 运行 `decay` 后自动获得项目变更模式量化数据 | 不知道哪些文件改动最频繁、哪些模块是变更热点 | 每次运行自动分析 git log，变更指标写入快照 |
| decay 开发者 | 新增一个数据源 | 新建采集函数 + 新建 DB 表 + 在 run.rs 手动调用 | 实现 Collector trait + 注册 1 行 |
| decay 开发者 | 处理采集器失败 | run.rs 中手动 match 错误 | 框架统一错误隔离 |

## 3. 核心假设

- **假设**: 自动扫描文件树 + 分析 git 历史 → 为评分维度提供数据基础
- **验证方式**: 对真实项目运行 `decay`，SQLite 快照中有文件结构数据和 git 历史数据
- **假设**: 将采集器抽象为统一 trait + 注册表 → 新增数据源的实现成本从"改 3+ 文件"降为"新增 1 个文件 + 注册 1 行"
- **验证方式**: 重构后现有两个采集器通过 trait 注册表统一调度，输出与重构前一致

## 4. 方案

### 文件结构扫描
- **Before**: 不知道项目有多少文件、目录多深、哪些文件过大 → **After**: 每次运行自动扫描文件树，结构指标写入快照

### git 历史分析
- **Before**: 不知道哪些文件改动最频繁、哪些模块是变更热点 → **After**: 每次运行自动分析 git log，变更指标写入快照

### 采集层插件化
- **Before**: scan.rs 和 git.rs 是独立函数，各自硬编码 DB 表，run.rs 手动调用并处理错误 → **After**: 每个采集器实现 Collector trait，通过注册表统一调度，DB 表由采集器自行管理

## 5. 验收标准

- 用户运行 `decay` → 应看到扫描项目文件树（排除 .git、target 等）
- 用户查看 SQLite 快照 → 应看到文件总数、目录总数、最大目录深度、各文件的路径和大小
- 用户对 1000+ 文件的项目运行 → 应看到扫描完成，耗时 <5 秒
- 用户运行 `decay` → 应看到分析项目 git 历史（默认最近 90 天）
- 用户查看 SQLite 快照 → 应看到每个文件的变更次数、最近修改时间、变更行数（churn）
- 用户对有 500+ commits 的项目运行 → 应看到分析完成，耗时 <10 秒
- `cargo test` 全部通过，`decay` 输出与重构前一致
- 新增采集器只需：创建文件、实现 Collector trait、在注册表中注册
- git 采集失败时其他采集器不受影响（错误隔离）

## 6. 排除项

- 语言级解析（AST 分析、函数计数等 → 语言增强）
- .gitignore 之外的自定义排除规则（后续版本按需添加）
- 作者维度分析（bus factor 等 → 后续版本）
- branch 分析（只看默认分支历史）
- 不新增任何采集器 — v3 各维度各自负责
- 运行时动态加载 — 编译期注册已满足需求
