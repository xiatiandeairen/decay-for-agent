# fragility 评分

## 1. 问题

项目脆弱性的核心信号来自变更模式——少数文件承受大量改动（热点）、高 churn 文件意味着不稳定、变更集中度高意味着风险集中。M2 采集了 git 历史数据，但需要将变更指标转化为一个可量化的脆弱度分数。

## 2. 目标用户

使用 decay 的 AI agent。agent 运行 `decay` 后获得 fragility 分数，用于判断哪些模块最脆弱、是否需要分离关注点。

## 3. 核心假设

**基于 git 变更模式计算 fragility 分数 → 用户能识别变更热点和脆弱模块，在脆弱性恶化前介入。**

验证方式：运行 `decay`，输出包含 fragility: XX/100 分数。

## 4. 方案

- **Before**: 知道哪些文件改动频繁但不知道整体脆弱度 → **After**: 输出 fragility 0-100 分，量化变更风险

## 5. 验收标准

- 运行 `decay` → 输出包含 fragility 分数（0-100）
- 变更越集中在少数文件、churn 越高 → 分数越低
- 对变更均匀分布的项目 → 分数 >80
- 无 git 历史时 → 输出 fragility: N/A（不报错）

## 6. 排除项

- 不包含作者维度分析（bus factor → 后续版本）
- 不包含文件结构评分（→ structural 维度）
- 不包含问题诊断和重构建议（→ M4）
