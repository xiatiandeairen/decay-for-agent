# git 历史分析

## 1. 问题

项目衰退的另一个核心信号来自变更模式——频繁修改的热点文件、高 churn 率的模块、单点集中的改动。这些信号隐藏在 git 历史中，人工翻 log 无法系统性提取。需要自动化分析 git 历史，量化变更维度的健康指标。

## 2. 目标用户

使用 decay 的 AI agent。agent 运行 `decay` 后自动获得项目变更模式的量化数据，无需手动分析 git log。

## 3. 核心假设

**自动分析 git 历史并采集变更指标 → 为 M3 的 fragility 维度评分提供数据基础。**

验证方式：对真实项目运行 `decay`，SQLite 快照中有 git 历史数据（变更频率、热点文件、churn 指标）。

## 4. 方案

- **Before**: 不知道哪些文件改动最频繁、哪些模块是变更热点 → **After**: 每次运行自动分析 git log，变更指标写入快照

## 5. 验收标准

- 运行 `decay` → 分析项目 git 历史（默认最近 90 天）
- SQLite 快照包含：每个文件的变更次数、最近修改时间、变更行数（churn）
- 对有 500+ commits 的项目运行 → 分析完成，耗时 <10 秒

## 6. 排除项

- 不包含作者维度分析（bus factor 等 → 后续版本）
- 不包含 branch 分析（只看默认分支历史）
- 不包含评分逻辑（→ M3）
