# 基础设施

## 1. 问题

### 痛点

decay-for-agent v1 需要从 Python MVP 迁移到 Rust 实现，从零建立项目结构、命令行入口和数据持久化层。没有这些基础设施，所有业务功能无法启动。

- **项目结构缺失**: 无 Rust 项目，无法开始任何功能开发
- **无命令行入口**: 项目可编译但用户无法通过 `decay` 命令与工具交互
- **无持久化存储**: 采集的数据无法保留，无法做跨时间点对比

### 影响范围

阻塞所有 v1 后续里程碑（数据采集、评分、处方）。所有用户交互依赖 CLI 入口，所有数据采集和趋势功能依赖快照存储。

### 为什么现在做

基础设施是所有功能的前置依赖。不做则后续功能开发、用户交互、数据持久化均无法启动。

## 2. 目标用户

| 角色 | 场景 | Before | After |
|------|------|--------|-------|
| AI agent (Claude Code) | 在项目上运行 `decay` 命令获取健康报告 | 无 Rust 项目存在，agent 无法执行任何 decay 操作 | `cargo build` 通过，CI 自动验证每次提交 |
| 开发者 / AI agent | 在终端中运行 `decay` 命令 | 项目可编译但无 CLI 入口，无法运行 decay | 运行 `decay --help` 输出帮助信息，为后续功能预留命令扩展点 |
| AI agent | 运行 `decay` 后采集数据自动持久化 | 采集数据无处写入，每次运行结果丢失 | 每次运行自动创建快照记录，数据持久化到 SQLite |

## 3. 核心假设

- **假设**: 建立标准 Rust 项目结构 + CI + CLI 框架 + SQLite 存储 → 后续功能可直接在此基础上迭代
- **验证方式**: `cargo build` 成功、CI green、`decay --help` 输出帮助信息、运行后 SQLite 数据库自动创建

## 4. 方案

### 项目初始化
- **Before**: 无 Rust 项目，无法开始 v1 开发 → **After**: `cargo build` 通过，CI 自动验证每次提交

### CLI 框架
- **Before**: 项目可编译但无命令行入口，用户无法运行 decay → **After**: 运行 `decay --help` 输出帮助信息，为后续功能预留命令扩展点

- **Before**: 采集数据无处写入，每次运行结果丢失 → **After**: 每次运行自动创建快照记录，数据持久化到 SQLite

## 5. 验收标准

- 用户运行 `cargo build` → 应看到编译成功，无 error
- 用户推送代码到仓库 → 应看到 CI pipeline 自动触发并 green
- 用户查看项目结构 → 应看到遵循 Rust 社区惯例（src/main.rs 或 src/lib.rs 入口）
- 用户运行 `decay --help` → 应看到包含工具名称和用法说明，退出码 0
- 用户运行 `decay --version` → 应看到版本号
- 用户运行 `decay`（无参数）→ 应看到帮助或提示信息，不 panic
- 用户运行 `decay` → 应看到 XDG data 目录下 `decay/snapshots.db` 自动创建
- 用户查看数据库 → 应看到快照元数据表（快照 ID、时间戳、项目路径）
- 用户多次运行 `decay` → 应看到每次生成独立快照记录

## 6. 排除项

- 任何业务逻辑实现（v1 后续里程碑负责）
- 子命令实现（采集、评分等功能在后续里程碑）
- `--json` 输出格式（→ M6）
- 具体的采集逻辑（→ 文件结构扫描 / git 历史分析）
- 快照清理策略（后续版本按需添加）
