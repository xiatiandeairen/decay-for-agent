# 可维护性维度

## 1. 问题

现有三个维度（structural / complexity / fragility）无法捕捉代码可维护性方面的腐败信号：重复代码在项目中蔓延、文件/函数过长难以理解、死代码积累增加认知负担。这些问题直接影响修改代码的成本和风险。

## 2. 目标用户

使用 decay 的开发者和 AI agent。可维护性评分让用户知道代码是否在变得越来越难维护。

## 3. 核心假设

**基于文件内容分析计算 maintainability 分数 → 用户能量化"代码多难维护"，识别重复代码蔓延和文件膨胀问题。**

验证方式：运行 `decay`，输出包含 maintainability 0-100 分数 + 诊断 + 处方。

## 4. 方案

- **Before**: 无法检测重复代码、长文件/函数、死代码
- **After**: maintainability 维度输出 0-100 分，诊断具体问题，提供重构处方

### 指标

| 指标 | 数据源 | 检测方式 |
|------|--------|---------|
| 重复代码比例 | 文件内容 | 行级哈希相似度检测（跨文件匹配连续重复行块） |
| 长文件比例 | 文件内容 | 行数统计，按语言适配阈值 |
| 长函数比例 | 文件内容 | 正则匹配函数定义，计算函数体行数 |
| TODO/FIXME 密度 | 文件内容 | 关键词 grep |

### 任务

| 任务 | 文档 | 进度 |
|------|------|------|
| maintainability tech | [tech](tech.md) | 0/0 |

## 5. 验收标准

- `decay` 输出包含 maintainability 分数（0-100）
- 重复代码多的项目 → 分数低 + 诊断指出重复块
- 长文件/函数多 → 分数低 + 诊断列出具体文件
- TODO/FIXME 多 → 信息级诊断
- 无需新的 collector（复用 file_scan 已写入 DB 的文件列表 + 直接读文件内容）

## 6. 排除项

- AST 级重复检测（行级哈希足够）
- 语言特定的死代码分析（需要编译器支持）
- 圈复杂度计算（归入 complexity 维度）
