# 质量保障维度

## 1. 问题
项目缺乏测试或测试质量低时，修改代码的信心不足。现有维度不检测测试覆盖和质量。

## 2. 核心假设
基于文件系统和内容分析检测测试比例和质量 → 用户能量化"测试保障有多强"。

## 3. 指标
| 指标 | 检测方式 |
|------|---------|
| 测试文件比例 | 文件名匹配 *_test.* / test_* / *_spec.* / tests/ 目录 |
| 测试/源码行数比 | 测试文件总行数 / 源码文件总行数 |
| 断言密度 | grep assert/expect/should/toBe 等 / 测试文件行数 |
| 测试命名规范 | 测试函数名是否包含 test/should/it/describe |

## 4. 验收标准
- quality_assurance 维度输出 0-100 评分 + 诊断 + 处方
- 无测试的项目 → 分数极低
- 测试少但有 → 分数中等 + 建议增加覆盖

## 5. 排除项
- 覆盖率百分比（需要运行测试工具）
- 变异测试
