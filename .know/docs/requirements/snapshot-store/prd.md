# 快照存储

## 1. 问题

decay 需要在每次运行时保存项目的健康数据快照，用于后续评分和趋势对比。当前没有持久化存储，采集的数据无法保留，也无法做跨时间点对比。需要一个结构化的存储层作为所有数据采集器的写入目标。

## 2. 目标用户

使用 decay 的 AI agent。agent 运行 `decay` 后，采集数据自动写入本地数据库，无需手动管理存储。

## 3. 核心假设

**提供 SQLite 快照存储 → 文件扫描和 git 分析的数据有统一的持久化目标，后续评分和趋势对比可直接查询历史快照。**

验证方式：运行 `decay` 后 XDG data 目录下 `decay/snapshots.db` 存在，包含快照表和正确的 schema。

## 4. 方案

- **Before**: 采集数据无处写入，每次运行结果丢失 → **After**: 每次运行自动创建快照记录，数据持久化到 SQLite

## 5. 验收标准

- 运行 `decay` → XDG data 目录下 `decay/snapshots.db` 自动创建
- 数据库包含快照元数据表（快照 ID、时间戳、项目路径）
- 多次运行 → 每次生成独立快照记录
- 文件扫描和 git 分析可通过统一接口写入同一快照

- 不包含远程存储或同步
