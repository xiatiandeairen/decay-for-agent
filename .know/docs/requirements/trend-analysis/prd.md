# 趋势分析

## 1. 问题

### 痛点

- **无法验证改进效果**: 单次运行的评分只反映当前状态，无法回答"比上次好了还是差了"。用户无法判断改动是否改善了项目健康。
- **无时间序列基础**: 只能对比当前和上一个快照（2 点对比），无法查询完整历史分数序列，后续趋势分析全部阻塞。
- **无变化率信息**: 只有原始分数序列没有变化率，用户和 agent 无法判断某个维度是在快速恶化还是缓慢改善。
- **无法检测突发恶化**: 只有 velocity 长期趋势，无法捕捉一次大规模重构或引入大量代码导致的分数骤降。
- **缺乏前瞻性**: velocity 和回归检测都是事后分析，无法提前知道"按当前趋势哪个维度会在什么时候跌破健康线"。
- **维度独立分析**: 各维度独立分析，错过联动模式。例如 complexity 上升时 maintainability 通常下降，但无法发现这种关联。
- **信息分散**: velocity、regression、forecast、correlation 分散在 Report 不同字段，无法一眼看到项目健康走向的整体画面。

### 影响范围

影响所有多次使用 decay 的项目。全部 8+ 个维度的趋势判断、突发恶化检测、未来预警、跨维度关联分析，以及所有消费趋势数据的场景（terminal/markdown/JSON/MCP）。

### 为什么现在做

评分和诊断已就绪，用户开始执行重构但无法验证效果。从基础对比到时间序列到变化率到回归检测到阈值预警到维度相关性到统一轨迹视图，是完整的时间维度分析体系。

## 2. 目标用户

| 角色 | 场景 | Before | After |
|------|------|--------|-------|
| AI agent | 重构后验证效果 | 每次运行只有当前分数，无法判断是否改善 | 自动对比上次快照，一眼看出变化方向 |
| AI agent | 基于 velocity 判断优先干预维度 | 只有原始分数，需人工比较 | 直接读取 velocity + 方向标签自动排序 |
| AI agent | 回归触发优先级提升 | 只有长期趋势，无法发现突发下降 | 回归标记自动关联相关 action |
| AI agent | 基于预警制定长期重构计划 | 只有事后趋势，无法预判 | 知道哪个维度将在几个快照后跌破阈值 |
| AI agent | 利用相关性优先处理根因维度 | 各维度独立分析，可能修复症状 | 发现联动模式，优先处理根因 |
| AI agent | 消费统一 trajectory 制定长期策略 | 需从多个字段组装趋势全貌 | 一个 trajectory 结构包含全部趋势数据 |
| 开发者 | 通过方向标签快速了解健康趋势 | 对比数字判断趋势 | 看到 ↑/↓/→ 标签一目了然 |
| 开发者 | terminal/markdown 查看回归警告 | 需人工对比相邻快照发现异常 | 醒目的回归警告 + 严重度分级 |
| 开发者 | 了解维度间的联动关系 | 无法看到维度间关联 | 直接看到强/中等相关性 |
| MCP server | 返回结构化 trajectory 数据 | 多个零散字段 | 统一 trajectory 字段 |

## 3. 核心假设

- **假设**: 自动对比上次快照 → 用户一眼看出变化方向
- **验证方式**: 第二次运行 `decay`，输出包含变化方向（↑↓→）和变化值
- **假设**: 复用现有 `dimension_scores` 表 + 新增时间序列查询 API → 后续趋势分析可直接消费时序数据
- **验证方式**: velocity 计算能直接调用 API 获取数据
- **假设**: 线性回归斜率 + 方向标签 → agent 和用户都能快速判断趋势方向和速度
- **验证方式**: ≥3 个快照时输出 velocity，方向标签准确反映分数走势
- **假设**: 相邻快照差值 > k × 历史标准差 → 准确标记回归，假阳性率 < 10%
- **验证方式**: k=2 时，基于历史快照序列回测假阳性率
- **假设**: 线性外推 + R² 置信度过滤 → 有意义的预警，避免噪声
- **验证方式**: R² > 0.7 时才预警，≥5 个快照数据验证准确性
- **假设**: Pearson 相关系数 + 强度分级 → 准确发现维度间联动模式
- **验证方式**: 已知关联（如 complexity↑ vs maintainability↓）被正确识别为强负相关
- **假设**: 聚合全部趋势数据为 Trajectory 结构 + overall direction → 快速判断项目健康走向
- **验证方式**: agent 能通过单个 trajectory 字段获取全部趋势信息

## 4. 方案

- **Before**: 每次运行只有当前分数 → **After**: 有历史时自动附带变化方向（↑↓→）
- **Before**: 只能对比当前和上一个快照 → **After**: 复用 `dimension_scores` 表，新增时间序列查询 API
- **Before**: 只有原始分数序列 → **After**: 线性回归斜率作为 velocity，映射为 Improving/Declining/Stable 方向标签
- **Before**: 只有 velocity 长期趋势 → **After**: 相邻快照差值标准差基线，超过 k×σ 标记回归（Moderate/Severe 两级）
- **Before**: 只有事后分析 → **After**: 线性外推预测阈值突破时间，R² > 0.7 过滤低置信预警
- **Before**: 各维度独立分析 → **After**: 所有维度对计算 Pearson 相关系数，|r| > 0.6 强相关，|r| > 0.4 中等
- **Before**: velocity/regression/forecast/correlation 分散展示 → **After**: Trajectory 聚合结构 + overall direction + 统一 "Health Trajectory" 段落

## 5. 验收标准

- 用户首次运行 `decay` → 应看到无趋势输出
- 用户第二次运行 `decay` → 应看到 ↑/↓/→ 和变化值
- 用户调用 `get_dimension_time_series()` → 应返回按时间排序的快照分数序列
- 用户有 ≥3 个快照 → 应看到 velocity 输出（方向标签 + 斜率）
- 用户分数大幅下降超过 k×σ → 应看到 Moderate 或 Severe 回归警告
- 用户有 ≥5 数据点 + R² > 0.7 + 斜率 < 0 → 应看到阈值预警
- 用户有 ≥5 共同数据点的维度对 → 应看到 |r| > 0.4 的相关性结果
- 用户有 ≥3 个快照 → 应看到 trajectory 字段，包含 overall direction
- 用户执行 `--json` → 应看到 time_series/velocities/regressions/forecasts/correlations/trajectory 字段
- 用户查看 markdown → 应看到 "Health Trajectory" 段落

## 6. 排除项

- 不包含历史图表 — 终端展示受限
- 不包含指定快照对比 — 自动对比上次即可
- 不做加速度计算（二阶导数）— 如有需要 v6+ 考虑
- 不做非线性外推 — 线性足够，ML 方法留 v6+
- 不做因果推断 — 相关不等于因果，需要领域知识判断
- 不做滞后相关（lag correlation）— v6+ 考虑
- 不删除原有零散字段 — 向后兼容
- 不做轨迹可视化 — CLI 文本足够，图形化留 v7+
