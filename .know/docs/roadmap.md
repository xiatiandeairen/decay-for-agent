## 1. 产品愿景

### 产品核心

- **定位**: 面向 AI agent 和 Claude Code 用户的项目健康监控工具
- **动机**: 项目在持续迭代中悄无声息地积累结构性债务，等到问题显现时清理成本已经很高；现有静态 lint 工具只看单次快照，无法追踪趋势和生成 agent 可消费的处方
- **长远愿景**: AI agent 在每次修改代码前自动感知项目健康趋势，主动避免引入衰退，并能执行结构化处方持续改善代码库

### 价值体系

| 层级 | 价值 | 衡量指标 |
|------|------|---------|
| **即时价值** | 一条命令获得 8 维度健康评分 + 分类诊断 + 可执行处方 | 扫描耗时（实测: <1s 中型项目）、issue 分类覆盖（实测: 8 类 A-H） |
| **累积价值** | 跨快照趋势追踪，回归自动检测并归因到 commit，改善效果可量化 | 快照序列长度（随使用增长）、回归检测准确率（目标值，待验证: 误报 <10%） |
| **战略价值** | 从"事后发现问题"到"事前预防衰退"，AI agent 从被动修复升级为主动守护 | 处方执行后分数变化（实测: --compare last 可量化）、预警准确率（目标值，待验证） |

### 核心问题

| 问题 | 发生频率 | 单次成本 | 影响面 | 现有应对 |
|------|---------|---------|--------|---------|
| 项目结构性债务积累，等发现时清理成本已经很高 | 每次迭代（估算: 日均数次代码变更） | 估算 10-30 分钟定位问题 + 未量化的回归风险（基于作者体感） | 所有维护中大型项目的开发者和 AI agent | 手动 code review，依赖经验判断，无量化趋势 |
| issue 数量多但不知道哪个该先修 | 每次健康检查（无 decay 前无此流程） | 估算 5-15 分钟筛选优先级（基于作者体感） | 同上 | 凭经验判断优先级，容易遗漏高影响问题 |
| 修了问题不知道有没有效果 | 每次重构后（估算: 周均 1-2 次） | 无法量化改善效果，降低修复动力 | 同上 | 无替代方案 |

### 目标用户

| 角色 | 典型场景 | Before | After | 预估提效 |
|------|---------|--------|-------|---------|
| 使用 Claude Code 的 AI agent | 在工作流中自动触发健康检查，获取结构化处方 | 无法感知项目健康状态，可能引入衰退（估算） | 通过 MCP/skill 自动获取诊断 + 处方，按优先级执行改善（已验证） | 待验证（预期: 减少 agent 引入的结构性衰退） |
| 使用 Claude Code 的独立开发者 | 迭代中定期检查项目健康，追踪改善趋势 | 依赖经验判断代码健康，无量化指标（估算） | 一条命令获得 8 维度评分 + 趋势 + 改善计划（已验证） | 待验证（预期: 单次检查节省 10-15 分钟人工审查） |

### 版本详情

#### v1 — CLI 核心闭环

- **战略意图**: 验证核心假设 — Rust CLI 能否在一条命令内完成项目健康检查闭环（扫描→评分→诊断→处方→趋势）
- **投入产出**: 投入初始开发 → 预期建立从扫描到处方的端到端管线（收益待后续版本量化）
- **优先级依据**: 闭环未验证前后续版本无意义；无外部依赖，可立即启动
- **风险与依赖**: 风险: Rust 生态中 git 分析库（git2）的跨平台兼容性未验证
- **成功指标**: `cargo build` 成功，`decay` 命令输出 3 维度评分 + issue 列表 + 处方建议 + 趋势对比，173+ tests 全部通过
- **核心价值**:
  1. 首次可以一条命令获取项目健康评分（structural/complexity/fragility）
  2. 首次可以获得诊断 issue 列表和处方建议
  3. 首次可以对比两次快照的分数变化
- **用户覆盖**: 作者 dogfood
- **核心指标**（N/A → v1）:

| 指标 | N/A | v1 | 变化 |
|------|-----|-----|------|
| 维度数 | 0 | 3 | ↑ 3 |
| 测试数 | 0 | 初始覆盖 | 新增基线 |
| 输出格式 | 0 | 1（terminal） | ↑ 1 |

#### v2 — Claude Code 集成

- **战略意图**: 让 decay 从"人手动运行"进化到"AI agent 自动调用"，验证 MCP + skill 集成可行性
- **投入产出**: 投入 MCP server + skill 开发 + 多格式输出 → 预期 agent 可程序化调用健康检查
- **优先级依据**: v1 闭环已验证，agent 集成是产品核心差异点；MCP 协议已稳定可用
- **风险与依赖**: 依赖 Claude Code MCP plugin 机制稳定性；依赖 @modelcontextprotocol/sdk
- **成功指标**: agent 通过 MCP 成功调用 `decay_check`，JSON/Markdown/quiet 输出格式正确
- **核心价值**:
  1. AI agent 可通过 MCP 自动调用健康检查
  2. `/decay` skill 提供零摩擦的会话内入口
  3. `--json` 输出 agent 可直接解析
  4. `--markdown` 输出可嵌入 PR 描述
  5. `--quiet` 输出 + exit code 可用于脚本判断
- **用户覆盖**: 作者 dogfood
- **核心指标**（v1 → v2）:

| 指标 | v1 | v2 | 变化 |
|------|-----|-----|------|
| 输出格式 | 1（terminal） | 4（terminal/json/markdown/quiet） | ↑ 3 |
| 接入方式 | 1（CLI） | 3（CLI/MCP/skill） | ↑ 2 |
| 维度数 | 3 | 3 | → |

#### v3 — 多维度评估

- **战略意图**: 从 3 维度扩展到 8 维度全面评估，建立 trait + registry 可扩展架构，支持不同项目类型的差异化评分
- **投入产出**: 投入架构重构 + 5 个新维度实现 → 预期覆盖代码质量的主要关注面
- **优先级依据**: 3 维度覆盖面不足，maintainability/reliability 等核心维度缺失；架构不扩展则每加维度都要改调度逻辑
- **风险与依赖**: 依赖 v1/v2 已完成；风险: 新维度的 grep 模式检测精度待验证
- **成功指标**: 8 维度全部输出分数，Dimension trait + Collector trait 可扩展注册，6 种项目类型自动识别
- **核心价值**:
  1. 新维度只需实现 trait + 注册，无需改调度逻辑
  2. 自动识别项目类型（CLI/WebService/Library/MobileApp/Monorepo/Generic），差异化权重
  3. 5 个新维度: maintainability/observability/quality_assurance/reliability/performance
- **用户覆盖**: 作者 dogfood
- **核心指标**（v2 → v3）:

| 指标 | v2 | v3 | 变化 |
|------|-----|-----|------|
| 维度数 | 3 | 8 | ↑ 5 |
| 项目类型 | 0（固定权重） | 6 | ↑ 6 |
| 架构可扩展性 | 硬编码调度 | trait + registry | 质变 |

#### v4 — 可执行处方

- **战略意图**: 从"文本建议"升级到"结构化 action"，让 agent 可直接解析并执行处方
- **投入产出**: 投入 Action schema + 处方引擎迁移 → 预期 agent 可消费 action 列表自动执行重构
- **优先级依据**: v3 的诊断已覆盖 8 维度，但处方仍是纯文本，agent 无法程序化消费
- **风险与依赖**: 依赖 v3 完成；风险: 位置精度（line_range）对部分维度需要扩展采集逻辑
- **成功指标**: 所有 8 维度的处方输出结构化 Action（含 ActionType/Priority/Effort/Target），action 可按优先级排序和去重
- **核心价值**:
  1. 7 种 ActionType（Split/Extract/Add/Remove/Replace/Move/Refactor）
  2. 4 级优先级 × 3 级 effort 分级
  3. Target 精确到文件 + line_range + symbol
  4. agent 可直接解析 action 列表执行重构
- **用户覆盖**: 作者 dogfood
- **核心指标**（v3 → v4）:

| 指标 | v3 | v4 | 变化 |
|------|-----|-----|------|
| Action 类型 | 0（纯文本处方） | 7 | ↑ 7 |
| 优先级分级 | 0 | 4（Critical/High/Medium/Low） | ↑ 4 |
| Effort 分级 | 0 | 3（Small/Medium/Large） | ↑ 3 |

#### v5 — 时序智能

- **战略意图**: 从 2 点对比升级到完整时间序列分析，提供 velocity/回归检测/预警/相关性等趋势能力
- **投入产出**: 投入时间序列查询 + 4 种分析算法 → 预期用户可看到完整的健康演进趋势
- **优先级依据**: v4 的处方已结构化，但趋势仅 2 点对比，无法发现渐进衰退或预测未来走向
- **风险与依赖**: 依赖 ≥3 个快照才能计算 velocity；风险: 线性回归在非线性趋势上预测偏差大
- **成功指标**: velocity 方向标签准确（≥3 快照），回归检测误报 <10%，预警在 R²>0.7 时触发，相关性需 ≥5 公共点
- **核心价值**:
  1. 衰退速度（velocity）：线性回归斜率 → Improving↑/Declining↓/Stable→
  2. 回归检测：均值 ± kσ 异常检测，Moderate/Severe 分级
  3. 阈值预警：线性外推预测何时跌破健康阈值
  4. 维度相关性：Pearson 相关系数发现维度间联动
  5. 轨迹报告：汇总所有趋势数据的 Trajectory 结构
- **用户覆盖**: 作者 dogfood
- **核心指标**（v4 → v5）:

| 指标 | v4 | v5 | 变化 |
|------|-----|-----|------|
| 趋势能力 | 2 点对比 | 完整时间序列（velocity/回归/预警/相关性/轨迹） | 质变 |
| 最小快照需求 | 2 | 3（velocity），5（相关性） | 细化 |

#### v6 — 智能诊断

- **战略意图**: 提升检测精度（消除误报）+ 建立 8 类问题分类体系，让 issue 列表从"平铺"升级到"分类优先"
- **投入产出**: 投入误报消除 + 分类引擎 → 预期用户看到的是"先修安全问题，再修模式问题"的优先序列
- **优先级依据**: v5 趋势能力就绪，但 issue 列表质量影响用户信任；test 函数中的 unwrap 误报是最常见的用户困惑
- **风险与依赖**: 依赖 v3 的 8 维度维度；风险: 分类规则覆盖度 — 新增 issue 类型可能未被分类
- **成功指标**: test 函数 unwrap 不再报 observability 问题，SQL injection 检测排除 error message 上下文，55+ 种 issue 全部归入 8 类
- **核心价值**:
  1. 消除已知误报（#[test] unwrap, error message SQL pattern）
  2. 8 类分类: MechanicalFix/PatternProblem/ArchitecturalDecision/SecurityCritical/ConventionDrift/ChronicDecay/ContextualException/Prevention
- **用户覆盖**: 作者 dogfood
- **核心指标**（v5 → v6）:

| 指标 | v5 | v6 | 变化 |
|------|-----|-----|------|
| 问题分类 | 0（平铺列表） | 8 类（A-H） | ↑ 8 |
| 误报率 | 无数据（未测量） | 已知误报消除（test unwrap, error SQL） | ↓ |

#### v7 — 闭环打磨

- **战略意图**: 评分校准 + 降噪 + 处方深化 + summary + compare 体验打磨，为 v8 的叙事化转变打基础
- **投入产出**: 投入体验打磨 → 预期输出质量和可读性显著提升（无新增量化指标）
- **优先级依据**: v1-v6 功能覆盖已完整，但输出体验需要打磨才能支撑 v8 的叙事化目标
- **风险与依赖**: 无已识别风险和外部依赖
- **成功指标**: summary 输出简洁准确，compare 输出清晰展示分数变化，filter pipeline 有效降噪
- **核心价值**:
  1. 评分校准：打分更准确反映项目实际健康状况
  2. Filter pipeline 降噪：减少低价值 issue
  3. Summary 输出：一行看到健康状态
  4. Compare 体验优化
- **用户覆盖**: 作者 dogfood
- **核心指标**（v6 → v7）:

| 指标 | v6 | v7 | 变化 |
|------|-----|-----|------|
| 测试数 | 无数据 | 173（实测: cargo test） | — |
| 源文件数 | 无数据 | 30+（实测） | — |

#### v8 — Impact-Driven Decay（影响驱动的衰退治理）

- **战略意图**: 翻转叙事 — 从"你有 87 个问题"到"修这 3 个问题能让你每次改代码省 15 分钟"，让用户感受改善而非被告知问题
- **投入产出**: 投入影响度量 + 处方追踪 + 叙事报告 → 预期用户从"被动接收问题清单"转为"主动追踪改善进度"
- **优先级依据**: v1-v7 建立了完整管线，但用户核心体验仍是"被告知哪里不好"；影响量化是提升处方可信度的关键
- **风险与依赖**: 风险: 影响评估不准确（"节省 X 分钟"可能不被信任）→ 保守估算，标注 estimated；git 共变分析在大项目可能慢 → 缓存 + 增量分析
- **成功指标**: 每个 issue 附带 Impact（coupled_files/review_burden/change_risk），`--compare last` 输出处方效果，改善计划分 3 阶段，报告首段为自然语言叙事
- **核心价值**:
  1. 影响度量化：每个 issue 附带"影响 N 个文件/每次改动多 X 分钟"
  2. 处方效果追踪：`--compare last` 自动对比执行前后维度变化
  3. 演进守护：自适应阈值 + 回退检测 + 归因到具体 commit
  4. 改善计划：分 3 阶段路线图（Quick Wins → Pattern Fix → Structural）
  5. 叙事化报告：首段自然语言摘要替代数字表格
  6. MCP 摘要优先 + skill 升级
- **用户覆盖**: 作者 dogfood
- **核心指标**（v7 → v8）:

| 指标 | v7 | v8 | 变化 |
|------|-----|-----|------|
| 影响度量 | 无 | 每个 issue 附带 Impact（coupled_files/review_burden/change_risk） | 新增 |
| 处方追踪 | 无 | --compare 关联处方效果 | 新增 |
| 改善计划 | 无 | 3 阶段路线图 | 新增 |
| 报告格式 | 指标表格 | 叙事化摘要 + 指标 | 质变 |
