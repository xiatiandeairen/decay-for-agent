# decay-for-agent 产品路线图

## 1. 产品愿景


|            |                                       |
| ---------- | ------------------------------------- |
| **解决什么问题** | 项目在持续迭代中悄无声息地积累结构性债务，等到问题显现时清理成本已经很高  |
| **给谁用**    | 使用 Claude Code 的开发者和 AI agent         |
| **核心差异**   | 不是静态 lint，而是跨快照的趋势追踪 + agent 可消费的重构处方 |


## 2. 版本规划

### v1 — Rust CLI 核心闭环 ✅


| 维度     | 标准                                          |
| ------ | ------------------------------------------- |
| **功能** | 单命令完成完整健康检查：采集 → 三维度评分 → 诊断 → 处方，有历史快照时附带趋势 |
| **质量** | 采集层 + 分析层有单元测试，核心路径有集成测试，`--debug` 支持调试     |
| **用户** | `decay` 一个命令即可使用，无需理解内部概念                   |


### v2 — Claude Code 集成层 ✅


| 维度     | 标准                                                          |
| ------ | ----------------------------------------------------------- |
| **功能** | agent 通过 MCP 自动调用 decay，用户通过 `/decay` 一键触发，sprint 完成后自动采集快照 |
| **质量** | MCP server 有协议测试，skill 有端到端验证                               |
| **用户** | 安装 plugin 后 `/decay` 即可使用，无需手动运行 CLI                        |


### v3 — 多维度评估 + 智能打分

| 维度     | 标准                                                          |
| ------ | ----------------------------------------------------------- |
| **功能** | 8 维度全面评估代码健康，按项目类型自适应打分，采集层插件化支持新数据源 |
| **质量** | 每个维度有独立单元测试，维度注册表有集成测试，分层打分有对比验证     |
| **用户** | 用户无感知升级，输出自动包含所有维度评分和诊断                   |

## 3. 当前版本

### 包含


| 能力              | 解决什么问题                                      |
| --------------- | ------------------------------------------- |
| **8 维度评估** | 3 个维度覆盖不全面 → 扩展为 structural / complexity / fragility / maintainability / observability & recoverability / quality assurance / reliability & security / performance |
| **分层打分** | 所有项目一套规则不合理 → 自动检测项目类型，适配阈值和权重 |
| **Dimension trait** | 新维度接入成本高 → trait 抽象 + 注册表，统一生命周期管理 |
| **采集层插件化** | 数据源硬编码 → trait 抽象，编译期注册新采集器 |
| **Composite score 升级** | 固定等权平均 → 按项目类型可扩展加权 |


### 排除

- 规则引擎（用户自定义策略检查 — 需求未验证）
- 新语言采集器（TypeScript / Go — 核心价值不依赖语言解析）
- 动态插件加载（运行时加载 — 编译期注册已满足需求）
- 用户自定义权重 UI（代码级可配已满足当前需求）
- 重复代码 AST 级检测（行级哈希足够，语言相关实现复杂度高）
- Documentation 维度（文档质量度量标准不明确）

## 4. 技术方向

### 维度体系


| # | 维度 | 关键指标 | 数据源 |
|---|------|---------|--------|
| 1 | structural | 文件数量、目录深度、顶层目录数 | 文件系统扫描 |
| 2 | complexity | 大文件比例、平均文件大小、最大文件 | 文件系统扫描 |
| 3 | fragility | 变更集中度、高频变更文件、最大单文件变更量 | git 历史 |
| 4 | maintainability | 重复代码比例、函数/文件长度分布、死代码比例 | 文件内容分析（文本相似度） |
| 5 | observability & recoverability | 日志/监控调用密度、错误处理覆盖率、panic/unwrap 密度 | 文件内容模式匹配 |
| 6 | quality assurance | 测试文件比例、测试/源码比率、断言密度 | 文件系统扫描 + 内容分析 |
| 7 | reliability & security | unsafe 代码比例、依赖漏洞信号、输入验证密度 | 文件内容 + 依赖文件解析 |
| 8 | performance | 嵌套循环密度、大数据结构拷贝模式、同步阻塞调用 | 文件内容模式匹配 |


### 架构升级

- **Dimension trait**：统一 collect → score → diagnose 生命周期，所有维度实现同一 trait
- **采集层插件化**：采集器抽象为 trait，编译期注册，新数据源不修改调度代码
- **分层打分**：ProjectProfile（项目类型自动检测）→ ScoreProfile（阈值 + 权重表）→ 各维度使用对应阈值评分
- **Composite score**：从固定等权平均升级为读取 ScoreProfile 权重的加权聚合

### 行业差异化

- **项目类型适配打分**：主流工具（SonarQube / CodeClimate / Codacy）均无此能力
- **可观测性维度**：静态分析工具完全空白的领域
- **测试质量评估**：超越覆盖率百分比，评估断言密度和测试结构

## 5. 风险与依赖


| 风险                          | 影响               | 缓解                              |
| --------------------------- | ---------------- | ------------------------------- |
| 模式匹配的准确率有限 | 新维度可能误报/漏报 | 保守阈值 + 明确标注为启发式分析 |
| 项目类型检测可能不准 | 打分规则不匹配 | generic fallback profile（等权重、中等阈值） |
| 维度数量翻倍影响性能 | 大项目分析变慢 | 按需启用维度，并行采集 |
| 现有维度重构可能引入回归 | v1/v2 用户受影响 | 重构后输出对比验证，确保一致 |
