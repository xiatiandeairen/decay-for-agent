# decay-for-agent 产品路线图

## 1. 产品愿景


|            |                                       |
| ---------- | ------------------------------------- |
| **解决什么问题** | 项目在持续迭代中悄无声息地积累结构性债务，等到问题显现时清理成本已经很高  |
| **给谁用**    | 使用 Claude Code 的开发者和 AI agent         |
| **核心差异**   | 不是静态 lint，而是跨快照的趋势追踪 + agent 可消费的重构处方 |

## 3. 当前版本

### 核心价值

**让处方从"建议"变成"指令"** — agent 读到 decay 输出后能直接执行重构，无需人工解读。

v1-v3 建好了"发现问题"的引擎（8 维度 + 分层打分 + 插件化架构）。v4 补上"解决问题"的最后一环：结构化、可执行的重构处方。

### 包含


| 能力                  | 解决什么问题                                          |
| ------------------- | ----------------------------------------------- |
| **Action Schema**   | 处方是自由文本，agent 无法解析 → 定义类型化的 action 结构            |
| **位置精度**            | 处方只说"文件 X 有问题" → 精确到行范围和函数名                     |
| **Action 生成器**      | 每个维度各自拼文本 → 统一的 action 生成接口，8 个维度各自实现           |
| **优先级排序**           | action 无序输出 → 按影响范围 × 修复成本排序，agent 知道先做什么       |
| **Agent 协议集成**      | MCP 返回文本 → 返回结构化 action 数组，agent 可直接消费           |


### 排除

- 自动执行 action（agent 决定是否执行，decay 只负责输出）
- AI 生成修复代码（action 描述"做什么"，不生成"具体代码"）
- 新语言采集器（TypeScript / Go — 核心价值不依赖语言解析）
- 跨项目聚合（v5+ 方向）
- 时间维度深化（衰退速度预测、回归检测 — v5 方向）

## 4. 技术方向

### Action Schema

```
Action {
    dimension: String,          // 来源维度
    action_type: ActionType,    // split | extract | add | remove | replace | move | refactor
    target: FileLocation,       // 文件路径 + 行范围 + 函数/模块名
    reason: String,             // 为什么需要这个 action
    priority: Priority,         // critical | high | medium | low
    impact: ImpactEstimate,     // 影响范围估算
}
```

### 维度 → Action 映射


| 维度                           | 典型 action 类型                                         |
| ---------------------------- | ---------------------------------------------------- |
| structural                   | split（拆分过大目录）、move（调整文件位置）                           |
| complexity                   | split（拆分大文件）、extract（提取模块）                           |
| fragility                    | extract（提取接口降低耦合）、refactor（分散变更集中度）                  |
| maintainability              | remove（删除死代码）、extract（消除重复）、split（拆分长函数）             |
| observability & recoverability | add（添加错误处理/日志）、replace（替换 panic 为 Result）            |
| quality assurance            | add（补充测试）、refactor（提升断言密度）                           |
| reliability & security       | replace（替换 unsafe）、add（添加输入验证）                       |
| performance                  | refactor（优化嵌套循环）、replace（避免不必要拷贝）                    |


### 行业差异化

- **结构化处方**：SonarQube / CodeClimate 只给文本建议，没有 agent 可消费的 action 格式
- **位置精度**：多数工具只到文件级，不到行范围 + 函数名
- **优先级排序**：现有工具按严重程度排序，不考虑修复成本

## 5. 风险与依赖


| 风险                  | 影响              | 缓解                                  |
| ------------------- | --------------- | ----------------------------------- |
| 位置精度依赖更深的代码分析       | 部分 action 位置不够精确 | 分层精度：有把握的给行级，否则给文件级，明确标注精度等级        |
| action 类型体系可能不够覆盖   | 新场景需要新类型        | action_type 设计为可扩展枚举，不硬编码           |
| 现有处方迁移可能丢失信息        | 部分文本描述难以结构化     | 保留 reason 自由文本字段，结构化部分是增量而非替代       |
| Agent 协议格式可能需要迭代    | 首版 schema 不一定最优 | 版本化 schema，向后兼容                     |


## 6. 后续版本方向（存档）

### v5 — Temporal Intelligence（时间维度深化）

- 衰退速度追踪：每个维度的分数变化速率
- 回归检测：新快照分数显著下降时告警
- 阈值预警：按当前趋势预测何时跌破健康阈值
- 维度间相关性：发现跨维度的联动模式
