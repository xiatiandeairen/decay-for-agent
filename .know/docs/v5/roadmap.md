# decay-for-agent 产品路线图

## 1. 产品愿景


|            |                                       |
| ---------- | ------------------------------------- |
| **解决什么问题** | 项目在持续迭代中悄无声息地积累结构性债务，等到问题显现时清理成本已经很高  |
| **给谁用**    | 使用 Claude Code 的开发者和 AI agent         |
| **核心差异**   | 不是静态 lint，而是跨快照的趋势追踪 + agent 可消费的重构处方 |


## 2. 版本规划

### v1 — Rust CLI 核心闭环 ✅


| 维度     | 标准                                          |
| ------ | ------------------------------------------- |
| **功能** | 单命令完成完整健康检查：采集 → 三维度评分 → 诊断 → 处方，有历史快照时附带趋势 |
| **质量** | 采集层 + 分析层有单元测试，核心路径有集成测试，`--debug` 支持调试     |
| **用户** | `decay` 一个命令即可使用，无需理解内部概念                   |


### v2 — Claude Code 集成层 ✅


| 维度     | 标准                                                          |
| ------ | ----------------------------------------------------------- |
| **功能** | agent 通过 MCP 自动调用 decay，用户通过 `/decay` 一键触发，sprint 完成后自动采集快照 |
| **质量** | MCP server 有协议测试，skill 有端到端验证                               |
| **用户** | 安装 plugin 后 `/decay` 即可使用，无需手动运行 CLI                        |


### v3 — 多维度评估 + 智能打分 ✅


| 维度     | 标准                                   |
| ------ | ------------------------------------ |
| **功能** | 8 维度全面评估代码健康，按项目类型自适应打分，采集层插件化支持新数据源 |
| **质量** | 每个维度有独立单元测试，维度注册表有集成测试，分层打分有对比验证     |
| **用户** | 用户无感知升级，输出自动包含所有维度评分和诊断              |


### v4 — Actionable Prescriptions（可执行处方）


| 维度     | 标准                                                    |
| ------ | ----------------------------------------------------- |
| **功能** | 处方从文本建议升级为结构化 action，包含精确文件位置和变更类型，agent 可直接消费执行重构 |
| **质量** | action schema 有类型约束，每个维度的 action 生成器有单元测试，端到端验证 action 可解析 |
| **用户** | 用户无感知升级；agent 消费 decay 输出后可自动执行处方，无需人工解读             |


### v5 — Temporal Intelligence（时序智能）


| 维度     | 标准                                                         |
| ------ | ---------------------------------------------------------- |
| **功能** | 跨快照的衰退速度追踪、回归检测、阈值预警，维度间相关性分析，从"当前健康"升级为"健康轨迹" |
| **质量** | 趋势算法有统计验证，预警准确率有回测，回归检测有假阳性率约束                     |
| **用户** | 用户看到的不只是分数，还有方向（↑↓→）、速度、预警，agent 可基于轨迹制定长期重构计划   |


#### 里程碑


| #   | 里程碑                                                                   | 验证点                                                          | 进度 |
| --- | --------------------------------------------------------------------- | ------------------------------------------------------------ | -- |
| M1  | **趋势引擎** — 快照序列存储 + 维度分数时间序列查询                                       | 给定 N 个快照，能查询任意维度的分数序列，支持时间范围过滤                               | ✅ |
| M2  | **衰退速度** — 每个维度计算分数变化率（velocity），标注加速/减速/稳定                           | 输出包含 velocity 数值和方向标签（↑↓→），连续 3 个快照即可计算                      | ✅ |
| M3  | **回归检测** — 新快照分数显著下降时自动标记回归                                            | 单次下降超过阈值触发回归标记，假阳性率 < 10%（基于历史快照回测）                         | ✅ |
| M4  | **阈值预警** — 按当前趋势线性外推，预测何时跌破健康阈值                                        | 输出 "按当前速度，{维度} 将在 {N} 个快照后跌破 {阈值}"，仅在趋势明确时预警               | ✅ |
| M5  | **维度相关性** — 发现跨维度联动模式（如 complexity ↑ 通常伴随 maintainability ↓）           | 输出维度间相关系数，标注强相关对，agent 可利用相关性优先处理根因维度                        | ✅ |
| M6  | **轨迹报告** — 整合速度、回归、预警、相关性为统一的 health trajectory 报告                      | 单次 decay 输出包含完整轨迹信息，MCP 返回结构化轨迹数据                           |    |
| M7  | **检测精度** — 区分测试/生产代码、消除误报、提升维度评分准确性                                      | observability 不再对 `#[cfg(test)]` 中的 unwrap 报警，reliability 不再对错误消息中的 format! 误报 |    |


## 3. 当前版本

### 核心价值

**从"当前健康"到"健康轨迹"** — 不只告诉你项目现在怎么样，还告诉你它往哪个方向走、走多快、什么时候会出问题。

v1-v3 建立了测量能力（8 维度 + 分层打分），v4 补上行动能力（结构化处方）。v5 补上预见能力：基于历史快照的趋势分析，让 agent 能制定长期重构策略而非只做当下修复。

### 包含


| 能力              | 解决什么问题                                                 |
| --------------- | ------------------------------------------------------ |
| **趋势引擎**        | 快照只做对比，没有时间序列 → 建立维度分数的时序存储和查询                        |
| **衰退速度**        | 只知道分数高低，不知道变化快慢 → 计算每个维度的 velocity，标注方向               |
| **回归检测**        | 分数下降要人工对比才能发现 → 自动检测显著下降并标记回归                         |
| **阈值预警**        | 问题暴露时已经很严重 → 基于趋势外推提前预警                               |
| **维度相关性**       | 维度独立分析，错过联动模式 → 发现跨维度相关性，定位根因维度                       |
| **轨迹报告**        | 信息分散在各维度 → 统一的 health trajectory 视图                    |
| **检测精度**        | 测试代码 unwrap 被误报为 observability 问题、错误消息 format! 被误报为 SQL 注入 → 区分测试/生产代码，精准匹配上下文 |


### 排除

- 自动触发重构（轨迹信息供 agent 决策，不自动执行）
- 机器学习预测模型（线性外推足够，ML 需要大量数据积累）
- 跨项目聚合（Portfolio Health — v6+ 方向）
- 可视化 dashboard（CLI + MCP 输出已满足需求）
- 自定义预警规则（内置规则先验证价值）

## 4. 技术方向

### 趋势引擎架构

```
SnapshotStore (现有)
  → TimeSeriesIndex { dimension → [(timestamp, score)] }
  → VelocityCalculator { 滑动窗口 + 线性回归斜率 }
  → RegressionDetector { 阈值 + 统计显著性检验 }
  → ThresholdForecaster { 线性外推 + 置信区间 }
  → CorrelationAnalyzer { 维度对 Pearson 相关系数 }
```

### 关键算法


| 模块       | 算法                                    | 最少数据量 |
| -------- | ------------------------------------- | ----- |
| 衰退速度     | 滑动窗口线性回归斜率                            | 3 个快照 |
| 回归检测     | 相邻快照差值 > k × 历史标准差（k 可配）              | 3 个快照 |
| 阈值预警     | 线性外推 + 置信区间，仅趋势 R² > 0.7 时预警          | 5 个快照 |
| 维度相关性    | Pearson 相关系数，|r| > 0.6 标注为强相关         | 5 个快照 |


### 数据量约束

快照数量不足时降级：
- < 3 个快照：仅显示当前分数和简单对比，不计算趋势
- 3-4 个快照：输出 velocity 和回归检测，不做预警和相关性
- ≥ 5 个快照：全功能

### 行业差异化

- **衰退速度量化**：SonarQube 有 "New Code" 概念但无速度追踪
- **阈值预警**：所有工具只报告当前状态，没有预测能力
- **维度相关性**：静态分析工具完全空白，没有跨指标联动分析

## 5. 风险与依赖


| 风险               | 影响             | 缓解                              |
| ---------------- | -------------- | ------------------------------- |
| 快照数量不足导致趋势不可靠    | 预警和相关性分析结果噪声大  | 数据量约束分级降级，不足时明确标注"数据不足"         |
| 线性外推对非线性衰退不准确    | 预警时间偏差大        | 标注预警为"按当前趋势"，附带置信区间             |
| 假阳性回归检测干扰用户信任    | 用户忽略真正的回归      | 可配 k 值，默认保守（k=2），低频误报优先于漏报      |
| 依赖 v4 action schema | 轨迹报告需要引用 action | v4 完成后再启动 v5，或轨迹报告先不引用 action   |


## 6. 后续版本方向（存档）

### v6+ — Portfolio Health（跨项目视角）+ Auto-fix

- 多项目健康聚合：组织级仪表盘
- 跨项目模式发现：所有项目的共性衰退趋势
- 团队级预警：组织范围内的健康阈值管理
- Auto-fix 能力：基于 action schema 自动生成修复代码，从"诊断+处方"升级为"诊断+处方+执行"
- fragility churn 智能化：区分"健康迭代"和"风险 churn"（如按代码复杂度加权），避免快速迭代期的 false positive
