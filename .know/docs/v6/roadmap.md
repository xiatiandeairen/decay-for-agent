# decay-for-agent 产品路线图

## 1. 产品愿景


|            |                                       |
| ---------- | ------------------------------------- |
| **解决什么问题** | 项目在持续迭代中悄无声息地积累结构性债务，等到问题显现时清理成本已经很高  |
| **给谁用**    | 使用 Claude Code 的开发者和 AI agent         |
| **核心差异**   | 不是静态 lint，而是跨快照的趋势追踪 + agent 可消费的重构处方 |


## 2. 版本规划

### v6 — Intelligent Diagnosis（智能诊断）

| 维度     | 标准                                                         |
| ------ | ---------------------------------------------------------- |
| **功能** | 问题自动分类（8 类），同类问题聚合为根因，按类别输出差异化解决策略，机械问题生成 patch，架构问题输出方案选项 |
| **质量** | 分类器有覆盖率测试（55 种 issue 全部映射），模式聚合有验证，patch 生成有可编译性验证 |
| **用户** | 看到的不再是 issue 列表，而是分类诊断报告：问题本质 + 解决思路 + 优先级排序 |


#### 问题分类体系

| 类型 | 名称 | 处理策略 | 输出形式 |
|------|------|---------|---------|
| A | 机械修复 | 模式匹配 → 生成 patch | Diff/Patch |
| B | 共性模式 | 聚合同类 → 识别根因 → 统一方案 | 根因分析 + 批量修复方案 |
| C | 架构决策 | 分析 trade-off → 输出选项 | 方案对比表 |
| D | 安全关键 | 精确定位 → 保守修复 → 必须审阅 | 修复建议 + 风险标注 |
| E | 规范漂移 | 检测不一致 → 识别主流模式 → 对齐建议 | 对齐计划 |
| F | 慢性恶化 | 结合 trajectory → 预防性预警 | 趋势预警 + 预防行动 |
| G | 语境例外 | 识别上下文 → 降级或静默 | suppress 建议 |
| H | 预防机制 | 推荐工具链/CI 配置 | 配置文件/脚本 |

| #   | 里程碑                                                    | 验证点                                              | 进度 |
| --- | -------------------------------------------------------- | --------------------------------------------------- | -- |

## 3. 当前版本

### 核心价值

**从"issue 列表"到"分类诊断报告"** — 不只告诉你有什么问题，还告诉你这些问题的本质（机械/模式/架构/安全/规范/慢性/误报/预防），以及每类问题的正确处理方式。

v1-v3 建立了测量能力（8 维度 + 分层打分），v4 补上行动能力（结构化处方），v5 补上预见能力（健康轨迹）。v6 补上理解能力：问题不只是被检测，还被分类、聚合、关联，agent 拿到的不是散列的 action，而是有组织的诊断报告。

### 包含

| 能力              | 解决什么问题                                                 |
| --------------- | ------------------------------------------------------ |
| **问题分类**        | 55 种 issue 一视同仁 → 按本质分为 8 类，每类有专属处理策略              |
| **语境识别**        | 测试代码 unwrap 被误报 → 识别上下文自动降级                        |
| **模式聚合**        | 10 个文件各报 1 条重复代码 → 聚合为 1 个"缺少公共抽象"根因             |
| **机械修复**        | action 只说"replace unwrap" → 生成可编译的 patch             |
| **预防配置**        | 修了又犯 → 生成 lint/CI 配置从源头阻止                           |
| **智能报告**        | issue 列表难以行动 → 分类报告有思路有优先级                         |
| **慢性预警**        | 分数未触发阈值但持续恶化 → 提前预警                                |

### 排除

- 自动执行修复（生成 patch，不 apply — v7+ 方向）
- 架构决策自动化（C 类输出选项，不替用户选）
- 安全问题自动修复（D 类只标记和建议，不自动改）
- 跨项目聚合（Portfolio Health — v7+ 方向）

## 4. 技术方向

### 分类引擎架构

```
Issues (v4 action list)
  + Trajectory (v5 velocity/regression/forecast)
  + ContextDetector (v6 M2: test/FFI/parser/builder markers)
  → IssueClassifier (M1: maps each issue to category A-H)
  → PatternAggregator (M3: groups same-category issues by root cause)
  → PatchGenerator (M4: generates diffs for category A)
  → PreventionRecommender (M5: generates CI/lint configs for category H)
  → DiagnosticReport (M6: structured output combining all above)
```

### 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 分类粒度 | 8 类 | 太细难维护，太粗失去差异化策略价值 |
| patch 格式 | unified diff | 通用、可审阅、可 apply |
| 语境检测范围 | 文件级 + 块级 | 文件名判断 + AST-lite 块追踪 |
| 聚合算法 | 基于 (category, dimension, action_type) 分组 | 简单有效，不需要 ML |

### 数据量约束

- 分类：无数据量要求，基于规则映射
- 模式聚合：≥2 个同类 issue 才聚合
- 慢性恶化：复用 v5 约束（≥3 快照 velocity，≥5 快照 forecast）

## 5. 风险与依赖

| 风险               | 影响             | 缓解                              |
| ---------------- | -------------- | ------------------------------- |
| 分类覆盖不全          | 新 issue 无分类     | 默认归类为 C（最保守），定期审计覆盖率         |
| patch 生成引入新问题   | 破坏编译或行为        | patch 必须通过 cargo check 验证       |
| 模式聚合过度          | 不相关的 issue 被合并  | 聚合键包含 dimension，跨维度不聚合         |
| 语境误判            | 生产代码被当作测试跳过   | 保守策略：只识别明确标记的上下文              |

## 6. 后续版本方向（存档）

### v7+ — Auto-execute + Portfolio Health

- 自动执行 A 类 patch（生成 → apply → 验证 → 提 PR）
- C 类架构决策辅助（调研行业标准做法，推荐方案）
- 多项目健康聚合：组织级仪表盘
- 跨项目模式发现：所有项目的共性衰退趋势
- 团队级预警：组织范围内的健康阈值管理
