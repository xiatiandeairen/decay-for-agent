# 阈值来源：经验判断 + 行业共识混合

## Why

decay 中有 40+ 个阈值常量，来源分三类：

### 统计学/行业共识（有依据）
- 回归检测 k=2.0 → 2σ 原则
- R² 门槛 0.7 → 统计学中等相关标准
- 嵌套深度 3 → 对应 O(n³) 认知复杂度（McCabe 复杂度研究）

### 经验判断（v7 校准后）
- 函数行数 60 → v7 从 50 调到 60，因为 50 误报率过高
- 重复率 15% → v7 从 5% 调到 15%，因为小项目天然重复率高
- churn 阈值 1000 → v7 从 500 调到 1000，过滤项目初期大量 churn 噪声

### 拍脑袋（无明确依据，但运行合理）
- review burden 分段（100/300/500/800 行 → 2/5/10/15/20 min）
- 耦合风险阈值（coupled ≥ 3 = high, ≥ 1 = medium）
- chronic decay 健康地板 70、预警线 80
- 语言过滤 10% 主语言门槛

## Rejected Alternatives

不做自适应阈值（per-project 学习）的原因：
- 需要大量历史数据（30+ snapshot）才有统计意义
- 增加用户理解成本（"为什么这个项目的标准和那个不一样"）
- v7 的手动校准已经覆盖了主要误报场景

## Constraints

- 修改阈值前应先检查 v7 校准记录（commit history），理解当初为什么调
- 同一维度的阈值要保持内部一致性（如 maintainability 的行数和函数数阈值应协调）
