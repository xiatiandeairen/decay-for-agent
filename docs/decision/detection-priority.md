## 1. 背景

**触发事件:** decay 把产品定位收敛为"AI 编程的复杂度退化早期警告器"后, 拆解出 AI 编程导致项目复杂度上升的 6 类典型现象, 需要决定 v0.1 聚焦做哪一类、哪些后置、哪些永久砍掉。

**约束条件:**
- 单人开发资源有限, v0.1 必须最小可信版本验证产品定位是否成立
- 现有工具（lizard/jscpd/SonarQube/CodeScene 等）已覆盖通用复杂度场景, 必须找到差异化空间
- AI 编程场景的特殊性是产品立足点, 选方向必须看是否对准 AI 留下的特异痕迹

**决策范围:** 仅决定 6 个候选方向的优先级 / 砍取, 不决定具体技术实现路线。

## 2. 决策

**我们决定:** A 类（函数级局部膨胀）作为 v0.1 唯一 P0; B 类（语义级重复扩散）作为 v0.4 的 P0; C/D 后置, E/F 永久砍掉。

A 是 AI 编程最高频且检测最确定的退化模式, 适合作为反馈环验证的最小切入点; B 是产品真正的差异化护城河（现有工具对"AI 改名重写"型重复全部漏检）, 但技术成本跳一档, 等 A 验证产品定位成立后再投入。

## 3. 候选方案

6 个候选方向跨四个维度的对比:

| 方向 | 检测可靠性 | AI 警觉价值 | 差异化 | 实现成本 | 综合 |
|------|----------|----------|-------|---------|------|
| **A. 局部膨胀** | 高 | 极高（每次改动直接看到 delta） | 中 | 中 | **P0** |
| **B. 重复扩散** | 中（看技术路线） | 极高（AI 最特有的退化） | 极高（无人做好） | 高 | **P0**（v0.4） |
| C. 死代码 | 中（动态语言上限） | 中（清理 ≠ 重构） | 低 | 高 | P2 |
| D. 防御堆叠 | 中 | 中 | 低 | 中 | P3（并入 A） |
| E. 抽象失配 | 低 | 中 | 中 | 极高 | 砍 |
| F. 文件爆炸 | 高 | 低 | 低 | 低 | 砍 |

### 方向 A: 函数级局部膨胀

> 检测每函数的行数 / 嵌套深度 / 圈复杂度 / 参数数等 metric 的退化

**优点:**
- AI 警觉价值极高: 每次改动可见 delta, 命中"修 bug = 加 if"这一最频繁的 AI 退化模式
- 检测可靠性高: 客观、可量化、确定（数得出来）, 适合作为客观裁判
- 实现成本中等: tree-sitter 已覆盖主流语言, metric 算法在业界成熟

**缺点:**
- 差异化中等: lizard / Clippy / ESLint 等已覆盖基础能力, 单靠这个不建壁垒
- 跨语言工程量线性增长: 每语言要适配函数边界 / 复杂度边定义
- 必须配合 diff 才有 AI 警觉价值（绝对值意义有限）

### 方向 B: 语义级重复扩散

> 检测函数级 / 语句块级的"语义重复", 即使变量名不同、结构略改也能识别

**优点:**
- AI 警觉价值极高: AI 编程最特有的退化（重写而非复用）, 现有工具检测不到
- 差异化极高: jscpd / PMD CPD 等只识别 token 级复制粘贴, 改名换汤型重复全部漏掉, 是真正的产品壁垒
- 跨语言天然单语言内问题, 不需要跨语言抽象

**缺点:**
- 实现成本高: 三条技术路线（token / AST 归一化 / embedding）各有死穴
- 检测可靠性受技术路线影响: token 级几乎无效, AST 归一化中等准, embedding 重而贵
- 性能挑战: O(N²) 函数对比, 大项目需 LSH / minhash 索引

### 方向 C: 死代码

> 未使用的 export / 调用图死分支 / 注释代码 / AI 留下的新旧版本并存

**优点:**
- 静态语言可达性强: Rust / Go / Java 等高置信度子集可识别
- AI 编程中常见: 新旧版本并存、注释代码、未引用的 helper

**缺点:**
- AI 警觉价值中等: 死代码是清理而非重构, 与"提前警觉"场景匹配度低
- 动态语言静态分析有理论上限: getattr / eval / 框架约定无法静态判断, 必有误报
- 实现成本高: 跨文件调用图、入口点识别、框架约定处理
- 差异化低: knip / vulture / cargo-machete 等单语言工具已成熟

### 方向 D: 防御性堆叠

> 嵌套 try / catch、重复 null check、过度参数化（boolean flag）

**优点:**
- 模式可识别: 嵌套 try/catch、同一变量重复 null check 等模式明确
- 实现成本中等: 模式匹配 + 密度计数即可

**缺点:**
- AI 警觉价值中等: AI 改完不一定能察觉, 告警价值有限
- 检测精度有限: "冗余 vs 必要"判断不可能 100% 准, 需要数据流分析
- 与 A 高度重叠: 防御代码必然增加嵌套和分支, 已被 A 的 metric 自然计入

### 方向 E: 抽象失配

> 单实现的 trait / interface、过度抽象（AbstractFactoryManager）、业务穿透到底层

**优点:**
- 设计层面有真实价值: 单实现 trait、Manager / Factory 命名密度等表面信号可识别

**缺点:**
- 检测可靠性极低: 本质是设计判断, 工具不知道哪是合理边界、哪是过度抽象
- 实现成本极高: 需要架构语境理解, 单实现 interface 误报严重（mock / 未来扩展 / 依赖反转）
- 差异化中等但价值不确定: 投入产出比最差, 容易做出"能跑但没人信"的指标

### 方向 F: 文件 / 模块爆炸

> 文件总数过多、目录深度过深、相似命名文件聚类（user.ts / userService.ts / userManager.ts）

**优点:**
- 检测可靠性高: 数文件、数行、聚类相似命名, 简单
- 实现成本低: 不需要 AST

**缺点:**
- AI 警觉价值低: "爆炸 vs 健康拆分"无客观界限, 信号弱
- 差异化低: SonarQube / CodeScene 等都有
- 与"复杂度上升"不强相关, 容易误报合理拆分

## 4. 影响

**正向影响:**
- v0.1 范围明确收敛: 单一目标（A 类反馈环验证）, ~8 天可交付（估算, 基于工作分解）, dogfood 路径清晰
- 资源聚焦: 不分散到 6 个方向每个都做不深; 先验证产品定位是否成立, 再决定差异化投入
- 砍掉的 E / F 永久剔除, 不再消耗规划注意力
- D 作为 A 的衍生信号观察, 不单独立项, 减少维护负担

**负向影响:**
- 短期与 lizard 等工具难差异化: v0.1 阶段 decay 在功能面上类似"带 diff 的 lizard"
- B 类延后到 v0.4: 产品壁垒建立的窗口期被推迟, 期间可能有竞品切入（目标值, 待验证: 当前未观察到聚焦"AI 风格重复"的工具）
- C 类用户痛点（清理 AI 留下的死代码）整体推后

**后续行动:**
- v0.1 实施过程中持续校验"AI 警觉价值"假设是否成立（dogfood 中至少 1 次工具捕捉退化的瞬间, 否则定位需重审）
- v0.1 完成后基于实证决定: v0.2 是否扩展语言, v0.4 是否启动 B 类
- D 类（防御堆叠）作为 A 的衍生指标观察, 单独立项需触发"A 类不足以覆盖防御退化"的实证
