# 局部膨胀 Case Map

> 日期: 2026-05-04
> 目的: 盘点“局部膨胀”在真实 AI 写代码场景里的典型 case，明确当前 4 个 metric 已覆盖什么、没覆盖什么，以及下一步最值得补什么。

## 1. 先下结论

当前 `decay` 的 4 个 metric:

- `nesting`
- `cyclomatic`
- `cognitive`
- `params`

覆盖的是 **局部膨胀里的结构复杂度骨架**，不是局部膨胀的全貌。

它们已经能抓到:

- 分支变多
- 嵌套变深
- 控制流更难读
- 函数签名变重

但还抓不好:

- 职责块堆叠
- 条件表达式膨胀
- 局部状态膨胀
- 错误处理/兼容逻辑吞噬主逻辑
- 平铺分支型“分发中心”函数

所以当前 4 个 metric 不是错，而是 **基础层**。

## 2. 当前 4 个 metric 各自擅长什么

### 2.1 `nesting`

最擅长抓:

- 一层套一层的逻辑
- 修一个 bug 再包一层 `if`
- 深层 guard / fallback / loop 嵌套

不擅长抓:

- 平铺分支很多，但不深
- 一个条件表达式特别复杂
- 函数职责块很多，但都平铺

### 2.2 `cyclomatic`

最擅长抓:

- 分支数量增加
- `if / else / match / ?` 路径变多

不擅长抓:

- 路径数量不多，但每个条件都很难读
- 职责混杂但分支不多
- 局部状态越来越乱

### 2.3 `cognitive`

最擅长抓:

- 深嵌套
- 复杂控制流组合
- guard / fallback / 分支混合导致的理解负担

不擅长抓:

- 大量职责块顺序堆叠
- 长布尔表达式
- orchestrator 型大函数但逻辑都比较平

### 2.4 `params`

最擅长抓:

- 函数签名越来越重
- 参数传递面越来越宽

不擅长抓:

- 绝大多数局部逻辑复杂化
- 状态膨胀
- 分支/嵌套/职责问题

而且它的行动性通常弱于前三个。

## 3. 真实 AI 场景里的典型局部膨胀 case

下面这些 case 更贴近真实 AI 写代码时的退化方式。

### Case A: 深层嵌套膨胀

表现:

- `if -> if -> if`
- `match -> if -> loop`
- 处理一个新边界再套一层

当前覆盖:

- `nesting`: 强
- `cognitive`: 强
- `cyclomatic`: 中
- `params`: 无

结论:

- **当前已覆盖得比较好**

### Case B: 平铺分支爆炸

表现:

- 很多 `else if`
- 很多 `match arm`
- 一个函数逐渐变成“规则分发器”

当前覆盖:

- `cyclomatic`: 强
- `cognitive`: 中
- `nesting`: 弱
- `params`: 无

问题:

- 当前能看出“路径多了”
- 但不容易表达“函数已经变成分发中心”

结论:

- **当前部分覆盖，但不够精准**

### Case C: 条件表达式膨胀

表现:

- `if a && (b || c) && !d && foo(x)`
- 条件本身很难读，但嵌套不深

当前覆盖:

- `cyclomatic`: 中
- `cognitive`: 中
- `nesting`: 弱
- `params`: 无

问题:

- 当前没有单独刻画“条件复杂度”

结论:

- **当前覆盖不足**

### Case D: 职责块堆叠

表现:

一个函数里顺序堆着:

- 参数处理
- 权限判断
- 主逻辑
- fallback
- logging
- 输出格式转换

当前覆盖:

- `cognitive`: 中
- `cyclomatic`: 中
- `nesting`: 弱
- `params`: 弱

问题:

- 这类问题很多时候不是“控制流复杂”
- 而是“函数承担了太多职责块”

结论:

- **当前覆盖不足**

### Case E: 局部状态膨胀

表现:

- 局部变量越来越多
- mutable state 越来越多
- 多次赋值、回填、状态切换

当前覆盖:

- `nesting`: 无
- `cyclomatic`: 弱
- `cognitive`: 弱
- `params`: 无

问题:

- 当前几乎没有直接指标看“状态表面复杂度”

结论:

- **当前明显缺失**

### Case F: 错误处理 / 兼容逻辑吞噬主逻辑

表现:

- 主逻辑很短
- 各种 fallback、兼容、特殊处理越来越多
- AI 为了“稳妥”不断往里补

当前覆盖:

- `cognitive`: 中
- `cyclomatic`: 中
- `nesting`: 中
- `params`: 无

问题:

- 当前能知道“变复杂了”
- 但看不出“主逻辑已经被淹没”

结论:

- **当前部分覆盖，但价值表达不够**

### Case G: Orchestrator / 调用扇出膨胀

表现:

- 一个函数调越来越多 helper / service
- 逻辑本身不复杂，但协调责任越来越重

当前覆盖:

- `nesting`: 弱
- `cyclomatic`: 弱
- `cognitive`: 弱
- `params`: 弱

问题:

- 当前几乎不看“外部交互面”

结论:

- **当前基本没覆盖**

## 4. 当前覆盖情况总表

| Case | 典型性 | 当前覆盖 | 主要依赖 metric | 结论 |
|---|---|---|---|---|
| 深层嵌套膨胀 | 高 | 高 | nesting, cognitive | 已较好覆盖 |
| 平铺分支爆炸 | 高 | 中 | cyclomatic, cognitive | 部分覆盖 |
| 条件表达式膨胀 | 高 | 低 | cyclomatic, cognitive | 覆盖不足 |
| 职责块堆叠 | 高 | 低 | cognitive | 覆盖不足 |
| 局部状态膨胀 | 中高 | 很低 | — | 明显缺失 |
| 错误处理/兼容吞噬主逻辑 | 高 | 中 | cognitive, cyclomatic | 部分覆盖 |
| 调用扇出膨胀 | 中 | 很低 | — | 基本缺失 |

## 5. v0.1 不该做什么

不要因为 case 多，就在 v0.1 一口气加十几个指标。

原因:

- 指标越多，噪音越大
- 当前主链路还没完全做稳
- 一次补太多会把“commit 前裁决”心智冲淡

所以 v0.1 不应该做的是:

- 继续堆很多弱信号 metric
- 为每种局部膨胀都发明一个新分数
- 在没有价值验证的前提下把输出搞成大报表

## 6. 最值得补的 3 个方向

如果只从“局部膨胀”内部补能力，我建议优先级是这样。

### 6.1 条件复杂度

为什么优先:

- 很常见
- 当前覆盖明显不够
- 比较符合 AI 经常补条件的行为

可以考虑的 proxy:

- 复杂布尔表达式节点数量
- 单条件里的逻辑操作符密度
- 条件 AST 深度

### 6.2 职责块堆叠 / 函数长度

为什么优先:

- 很多真实膨胀不是“分支深”，而是“职责多”
- 函数长度 / statement 数是粗糙但高价值的 proxy

可以考虑的 proxy:

- statement 数
- body 行数
- 不同 block 段数量

### 6.3 局部状态复杂度

为什么优先:

- AI 很容易写出“小状态机式”函数
- 这是当前几乎完全缺失的面向

可以考虑的 proxy:

- 局部变量数
- mutable 绑定数
- 赋值点数量

## 7. 如果只允许补一个

如果 v0.1.x / v0.2 只允许补一个新方向，我建议优先补:

> **函数长度 / statement 数**

原因不是它最优雅，而是它最现实:

- 易实现
- 好解释
- 能补职责块堆叠
- 也能部分补“主逻辑被兼容逻辑吞噬”的问题

它比继续纠结 `params` 更可能提供增量价值。

## 8. 最终判断

当前 4 个 metric 已经抓住了局部膨胀里最经典的“结构复杂化骨架”，尤其是:

- 嵌套变深
- 分支变多
- 控制流更难读

但如果问“更典型、更贴近真实 AI 写代码的局部膨胀 case 是否还有很多没覆盖好”，答案是:

- **有，而且不少**

最值得重视的是:

1. 条件表达式膨胀
2. 职责块堆叠
3. 局部状态膨胀
4. 错误处理/兼容逻辑吞噬主逻辑

所以，当前 4 个 metric 可以视为:

- **局部膨胀 v0.1 的基础骨架**

而不是:

- **局部膨胀问题的完整地图**
