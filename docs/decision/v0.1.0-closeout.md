# v0.1.0 文档收尾 Decision Record

## 1. Background

**Triggering event:** v0.1.0 功能闭环已经跑通，但项目文档在三件事上不可靠：旧文档数量过多、命令语义曾与源码变化不同步、部分 metric 承诺曾早于实现事实。

**Constraints:**
- v0.1.0 没有外部用户，CLI、SQLite schema、阈值和文档结构都可以破坏式调整。
- `cargo test` 和 `cargo clippy -- -D warnings` 通过只能证明功能质量，不能证明 `decay diff` 会改变 commit 决策。
- 文档必须遵循 `know write` 的类型和路径：roadmap、prd、arch、decision、ops、milestone。
- 当前仓库事实以源码、测试、本地命令输出和已保留文档为准；不准确结论必须删除或标注 no data。

**Decision scope:** 本决策只处理 v0.1.0 收尾时“哪些上下文应进入长期文档、哪些历史材料应删除”的文档治理问题；不决定新的 metric、CI 集成、多语言支持或 baseline UX 方案。

## 2. Decision

**We decided:** v0.1.0 收尾文档只保留能支撑后续产品判断的高信号上下文，并按 `know write` 规范组织；过程日志、过期计划、旧命令假设和未实现 metric 承诺不留在当前 docs 工作树。

核心理由是：下一阶段真正要验证的是 `decay diff` 是否改变 commit 决策。文档如果混入旧过程材料或未验证结论，会让后续工作把“功能完成”误读成“产品成立”。

## 3. Alternatives

### Alternative A: 保留所有历史文档作为 archive

**Pros:**
- 最大化保留过去计划、实验和讨论的可见性。
- 不需要立即判断哪些内容仍然准确，短期迁移成本低。

**Cons:**
- 旧命令语义和旧 metric 承诺会继续出现在搜索结果里，增加误读概率。
- 过程日志会稀释当前事实，后续 agent 需要花更多时间判断哪些内容仍然有效。

### Alternative B: 只保留 know 规范下的当前事实文档

**Pros:**
- 当前事实有稳定入口：`docs/roadmap.md`、PRD、arch、decision、ops、milestone。
- 文档能直接服务下一阶段 dogfood，不再把旧过程记录当成当前决策依据。

**Cons:**
- 旧实验细节不再在 docs 工作树中直接可见，需要通过 git history 考古。
- 需要持续维护文档与源码一致，否则精简后的文档一旦错，会更容易被误认为权威事实。

### Alternative C: 暂不整理文档，等 dogfood 后再处理

**Pros:**
- 避免在产品仍未验证时投入文档整理成本。
- 保留所有上下文，短期内不会误删可能有用的材料。

**Cons:**
- dogfood 会从混乱上下文开始，难以区分当前事实、旧计划和愿望。
- 后续容易继续围绕 `doctor` finding 或新增 metric 展开，而不是验证 `diff` 的 commit 决策价值。

## 4. Impact

**Positive impact:**
- 后续进入项目时，可以从规范路径读取当前产品状态和下一步验证方法。
- `diff` 是核心价值路径、`doctor` 是辅助体检、active metric 只有 6 个，这些事实被集中保留。
- dogfood 必须记录 `fixed / ignored / noise`，避免用主观印象判断产品是否成立。

**Negative impact:**
- 删除的旧验证材料不再能直接从 docs 目录浏览。
- 文档治理变成显式责任：命令行为、metric registry、README、PRD、arch 必须同步更新。

**Follow-up actions:**
- 按 `docs/ops.md` 记录真实 dogfood case。
- 在 `docs/roadmap.md` 和 `docs/milestones/m2.md` 中只用真实数据更新 v0.1.x 结论。
- 在没有 fixed case 前，不把新增 metric、JSON/CI 集成或多语言支持列为当前阶段工作。
