# M2 — 验证 `diff` 是否改变 commit 决策

## 1. Goal

### Definition of Done

完成一轮真实 AI 编程 dogfood，并把每个 `decay diff` 命中归类为 `fixed`、`ignored` 或 `noise`。

### Value Target

作者能判断 `decay diff` 是否真的帮助发现本来会漏掉、且值得返工的复杂度退化。

### Key Hypothesis

- **Hypothesis**: `decay diff` 能在真实 AI 编程任务中产生至少 1 个作者原本没注意到、且愿意返工的 fixed case。
- **Type**: business value
- **Validation method**: 按 `docs/ops.md` 记录 dogfood case；出现 ≥1 个 fixed case 则继续 v0.1.x，否则重新评估产品切口。

## 2. Plan

### Planned Deliverables

- data: dogfood case 记录，字段包含 task、baseline、command、result、case_type、function、metric、human_judgment、action_taken。
- doc: `docs/roadmap.md` 中 v0.1.x 的真实结果更新。

### Go/No-Go Criteria

| Dimension | Criterion |
|-----------|-----------|
| Functional completeness | 至少完成 1 轮真实 AI 编程任务的 baseline + diff 记录（team agreement） |
| Quality | 每个命中都归类为 fixed / ignored / noise（team agreement） |
| Performance and stability | no data（M2 不验证性能） |
| Security | 不上传源码或 dogfood 记录到外部服务（team agreement） |
| Maintainability | 不新增 metric，除非已有 fixed case 证明当前链路有价值（team agreement） |

### Estimated Effort

no data（取决于真实 AI 编程任务数量）

### Decision Path

- **Pass**: 若出现 ≥1 个 fixed case，优先评估 baseline 使用摩擦和输出裁决语义。
- **Fail**: 若连续 dogfood 没有 fixed case，暂停新增功能，重新评估产品切口或阈值/scope。

### Risks

Dependencies: 作者持续按流程使用 baseline/diff；Risks: 命中可能主要是旧债、无关问题或作者不愿处理的问题。
