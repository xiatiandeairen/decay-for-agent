# v0.1 dogfood 价值验证方案

> 日期: 2026-05-04
> 目的: 验证当前 `Wave 1 + scope/role` 的局部膨胀链路，是否真的改变 commit 前决策，而不是只证明“工具能跑”

## 1. 验证目标

这轮 dogfood 只回答 3 个问题:

1. 当前 `check / diff / hotspots` 是否真的改变 commit 前决策
2. 当前告警里，有多少是高行动性信号，有多少是可忽略噪音
3. 在不引入 `Wave 2` 的前提下，当前局部膨胀方案是否已经足够支撑 v0.1

不回答的问题:

- `mutable_bindings` 是否值得做
- 重复扩散是否应该进入 v0.2
- 多语言是否要提前

这些都放后面。

## 2. 当前被验证对象

本轮只验证当前已经实现的主链路:

- `scope`
  - `prod`
  - `all`
- `role` 收敛
  - `tests/examples/benches/fixtures`
  - `testutil.rs`
  - `#[test]`
  - `#[cfg(test)]`
  - `mod tests`
- 局部膨胀指标
  - `nesting`
  - `cyclomatic`
  - `cognitive`
  - `params`
  - `statement_count`
  - `max_condition_ops`

默认使用口径:

- 日常裁决用 `prod`
- 只在怀疑误伤或遗漏时，再人工对照 `all`

## 3. 验证周期

建议周期: `3-5` 个真实开发日。

理由:

- 少于 3 天，样本太容易只覆盖一两类改动
- 超过 5 天，如果还没有出现明确决策改变案例，说明当前价值可能本来就不够强

最小样本要求:

- 至少 `10` 次真实 `check` 或 `diff`
- 至少 `5` 次 AI 参与的实际代码改动
- 至少 `3` 次出现告警的 session

## 4. 使用方式

每次真实开发 session，按下面顺序执行:

1. AI 改完一波代码
2. commit 前运行 `decay check`
3. 如果需要看快照间变化，再运行 `decay diff`
4. 只记录“是否改变决策”，不记录所有命令输出

推荐命令:

```bash
decay check
decay diff
```

如果怀疑默认 `prod` 过滤过头，可做一次对照:

```bash
decay hotspots --scope all
```

但 `all` 只用于诊断，不作为日常主口径。

## 5. 每次必须记录的字段

每个 session 只记下面 8 项，避免记录负担太高:

| 字段 | 含义 |
|---|---|
| 日期 | 绝对日期 |
| 仓库/分支 | 发生在哪个项目 |
| 改动类型 | `bugfix` / `feature` / `refactor` / `test-only` |
| AI 参与度 | `high` / `medium` / `low` |
| 是否有告警 | `none` / `low` / `high` |
| 最关键告警 | 哪个函数、哪个 metric |
| 是否改变决策 | `no` / `inspect more` / `rework now` |
| 作者结论 | 这次告警是价值、噪音，还是低行动性提示 |

## 6. 判定分类

每次告警都必须落到下面 4 类之一:

### A. `价值命中`

定义:

- 原本没意识到这次改动已经让函数变坏
- 被 `decay` 提醒后改变了动作
- 包括:
  - 立刻返工
  - 暂缓 commit
  - 补一轮更仔细 review

这是最重要的正样本。

### B. `确认型价值`

定义:

- 本来已经隐约觉得这段代码变胖了
- `decay` 只是把这个直觉客观化
- 最终仍帮助你更快做决定

这类价值弱于 A，但仍算有效价值。

### C. `低行动性提示`

定义:

- 告警 technically 合理
- 但不值得现在处理
- 例如:
  - 已知临时实现
  - 当前改动窗口不适合重构
  - 风险存在，但业务优先级更高

这类不算噪音，但也不算强价值。

### D. `噪音`

定义:

- 告警不值得关心
- 或明显来自错误扫描对象
- 或解释不成立

例如:

- 你明确不认为这段主维护面代码存在实际退化
- 或 `prod` 仍混进了不该看的角色代码

## 7. v0.1 通过标准

本轮 dogfood 通过，不看“总告警数”，只看下面 3 条:

### Hard gate

至少出现 `1` 次 `A. 价值命中`。

也就是:

> 工具抓到了作者/AI 原本没意识到的退化，并且真的改变了动作。

没有这条，v0.1 核心假设就不成立。

### Soft gate 1

`A + B` 合计至少 `3` 次。

说明当前工具不只是“偶然抓中一次”，而是开始稳定参与决策。

### Soft gate 2

`D. 噪音` 不高于全部有告警 session 的 `40%`。

超过这个比例，就说明信任成本开始偏高。

## 8. Stop conditions

出现下面任一情况，可以提前停止 dogfood，直接回头修产品:

1. 连续 `5` 次有告警 session 都落入 `C` 或 `D`
2. `prod` 仍频繁混入明显非主维护面代码
3. 作者开始因为不信任输出而不再愿意跑 `check/diff`

这三条里，第三条最危险。

因为一旦用户不愿意跑，说明产品链路已经断了。

## 10. 周期结束后的汇总模板

周期结束后，只汇总下面 6 个数字:

| 指标 | 数值 |
|---|---:|
| 总 session 数 |  |
| 有告警 session 数 |  |
| `A` 数 |  |
| `B` 数 |  |
| `C` 数 |  |
| `D` 数 |  |

然后只回答 3 个问题:

1. 是否出现了 `A`
2. `A + B` 是否达到 `3`
3. `D / 有告警 session` 是否超过 `40%`

## 11. 结果如何驱动下一步

### 情况 1: 通过

条件:

- 有至少 `1` 次 `A`
- `A + B >= 3`
- 噪音比例可接受

动作:

- 可以认为 v0.1 主链路成立
- 再决定是否进入 `Wave 2`

### 情况 2: 有用，但不稳

条件:

- 有 `A` 或 `B`
- 但 `C/D` 明显偏高

动作:

- 先继续降噪/提解释
- 不急着加新 metric

### 情况 3: 未成立

条件:

- 没有 `A`
- `B` 也很少
- 或噪音压过价值

动作:

- 不开 `Wave 2`
- 先重审 v0.1 的产品边界

## 12. 当前建议

当前最合理的执行顺序是:

1. 用这份方案做一轮 `3-5` 天 dogfood
2. 只记录真实决策改变，不记录大而全命令输出
3. 周期结束后再决定:
   - v0.1 是否成立
   - `Wave 2` 是否值得开

一句话说:

> **现在最缺的不是新检测能力，而是“当前能力是否真的改变行为”的证据。**
