# decay 验证矩阵

> 目标: 从 **实用性 / 价值性 / 可用性** 三个维度验证 `decay`，避免只停留在“能跑”层面。

## 1. 实用性

| 编号 | 场景 | 操作 | 预期 |
|---|---|---|---|
| U1 | 真实 Rust 项目建基线 | 在真实项目根目录运行 `decay init` | 成功扫描、成功落库、输出摘要，不崩溃 |
| U2 | 热点查看 | 运行 `decay hotspots` | 输出当前超阈值函数列表，排序稳定，热点可人工解释 |
| U3 | 无改动检查 | 基线后立即运行 `decay check` | 输出 clean，不误报 regression |
| U4 | 受控复杂化改动 | 对一个低复杂函数人工增加嵌套，再运行 `decay check` | 抓到对应函数的 degradation |
| U5 | 历史快照比较 | 连续 `init` 两次，再运行 `decay diff` | 无改动时 clean；有改动时能正确对齐并输出 |
| U6 | 排除非核心目录 | 使用根 `.gitignore` 或 `--exclude` 排除 examples/generated 等目录 | 排除目录不进入扫描结果 |

## 2. 价值性

| 编号 | 场景 | 操作 | 预期 |
|---|---|---|---|
| V1 | 基线热点可信度 | 抽样阅读前 5-10 个热点函数源码 | 大部分热点有明显复杂控制流，不是随机噪音 |
| V2 | 误报识别能力 | 找一个看上去不该高分的函数，对照源码和 metric 解释 | 能发现算法缺陷或阈值失真并定位根因 |
| V3 | diff 的动作价值 | 受控改动后看 `check/diff` 输出 | 输出能直接支持“是否立即重构”的决策 |
| V4 | 与产品定位一致性 | 首次使用和日常使用各跑一遍 | 用户第一感知是 baseline/check，而不是纯热点报告 |

## 3. 可用性

| 编号 | 场景 | 操作 | 预期 |
|---|---|---|---|
| X1 | 首次使用 | 在空 DB 下运行 `decay init` | 文案能说明已建 baseline，下一步该跑什么 |
| X2 | 无 baseline 误用 | 直接运行 `decay check` / `decay diff` | 给出明确引导，而不是报错或沉默 |
| X3 | 命令职责清晰 | 分别运行 `init/check/diff/hotspots --help` | 命令语义互不混淆 |
| X4 | 默认入口合理 | 直接运行 `decay` | 等同日常主入口 `check` |
| X5 | 可控排除 | 运行 `hotspots --exclude ...` | 用户能主动压掉噪音目录 |

## 4. 推荐执行顺序

1. 先跑 `cargo test`
2. 在真实项目跑 `decay init`
3. 跑 `decay hotspots`
4. 跑 `decay check`
5. 注入受控回归，再跑 `decay check`
6. 必要时再跑 `decay diff`
7. 记录价值判断、误报、根因和修复建议
