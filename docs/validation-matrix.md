# decay 验证矩阵

> 目标: 按当前产品思路验证两件事
> 1. `decay` 扫描的对象是否真的是“主维护面源码”
> 2. `decay` 在“局部膨胀”上是否真的提供了 commit 前裁决价值
>
> 补充:
> - 如果要看完整用户需求侧结论，见 [docs/value-validation.md](/Users/taoxia/Workspace/self/skills/decay/docs/value-validation.md)
> - 如果要看扫描范围和局部膨胀的设计定义，见 [docs/decision/scan-scope-and-local-bloat-value.md](/Users/taoxia/Workspace/self/skills/decay/docs/decision/scan-scope-and-local-bloat-value.md)

## 1. 验证原则

- 不再把“命令能跑”当成主要结论。
- 先验证扫描对象是否对，再验证 metric / diff 是否有意义。
- 先看 `check / diff` 是否支持真实决策，再看 `hotspots` 是否有辅助价值。
- 明确区分:
  - **当前实现行为**
  - **目标产品定义**
- 如果当前行为和目标定义不一致，验证结果必须把偏差写出来，不能混过去。

## 2. 当前目标定义

按当前思路，`decay` 的产品定义应该是:

- 默认关注 **主维护面源码**
- 核心价值是 **AI 改完代码后的 commit 前局部膨胀裁决**
- `hotspots` 是辅助视图，不是主价值链路

因此，验证必须围绕两条主线展开:

1. **扫描对象验证**
2. **局部膨胀价值验证**

## 4. 主线 A: 扫描对象验证

### 4.1 核心验证问题

| 编号 | 问题 | 为什么重要 |
|---|---|---|
| S1 | 当前默认扫描，是否混入了非主维护面代码？ | 如果混入，主链路信号会被污染 |
| S2 | 用户是否必须知道大量 `--exclude` 细节，才能得到可信结果？ | 如果必须，产品门槛过高 |
| S3 | 排除 `examples/tests/fixtures` 后，结果质量是否显著提升？ | 如果提升明显，说明默认扫描对象定义有问题 |
| S4 | 当前实现和目标定义之间的差距是否被清晰记录？ | 否则团队会误以为问题已解决 |

### 4.2 执行场景

| 编号 | 场景 | 执行方式 | 关注点 | 结论类型 |
|---|---|---|---|---|
| S1-A | 自测仓库默认热点 | 在 `decay` 自测副本执行 `decay hotspots` | 是否出现 `tests/fixtures` 等辅助代码 | 当前行为 |
| S1-B | 自测仓库排噪后热点 | 执行 `decay hotspots --exclude tests --exclude examples` | 结果是否更接近主维护面 | 当前行为 |
| S2-A | `ripgrep` 默认 / 排噪对比 | 对比 `hotspots` 与 `hotspots --exclude examples` | 热点数和信号密度变化 | 当前行为 |
| S3-A | 无 baseline + clean + 退化链路 | 用 `check/init/diff` 跑一遍 | 主链路是否依赖排除策略才能可信 | 当前行为 |
| S4-A | 设计定义核对 | 对照 [docs/decision/scan-scope-and-local-bloat-value.md](/Users/taoxia/Workspace/self/skills/decay/docs/decision/scan-scope-and-local-bloat-value.md) | 当前实现是否已经满足 `prod scope` 定义 | 目标差距 |

### 4.3 通过标准

不是“默认扫描完全正确”，而是:

- 能明确识别哪些目录正在污染主链路
- 能明确证明排噪后结果是否变好
- 能明确说明当前默认扫描还不等于目标中的 `prod scope`

也就是说，扫描对象验证的产出应该是 **清晰边界**，不是粉饰太平。

## 5. 主线 B: 局部膨胀价值验证

### 5.1 核心验证问题

| 编号 | 问题 | 为什么重要 |
|---|---|---|
| B1 | `check` 是否能在 commit 前快速给出裁决？ | 进不了主工作流就没有价值 |
| B2 | `diff` 是否能支撑“这次改动值不值得返工”的判断？ | 这是核心承诺 |
| B3 | `hotspots` 是否只是辅助，而不是主价值？ | 避免产品心智跑偏 |
| B4 | 当前验证是否已经证明“抓到了真实会漏掉的问题”？ | 这是价值深度的关键分界线 |
| B5 | 当前哪些点仍然让局部膨胀价值打不穿？ | 为下一轮产品迭代定优先级 |

### 5.2 执行场景

| 编号 | 场景 | 执行方式 | 关注点 | 结论类型 |
|---|---|---|---|---|
| B1-A | clean check | baseline 后立刻执行 `decay check` | 是否快速、稳定、无误报 | 当前行为 |
| B2-A | 受控退化 check | 注入 5-6 层嵌套后执行 `decay check` | 是否精确指出函数和跨阈值 metric | 当前行为 |
| B2-B | 受控退化 diff | 追加 baseline 后执行 `decay diff` | 是否与 `check` 指向同一退化 | 当前行为 |
| B3-A | 初次 `init` 与 `hotspots` 对比 | 对比首次基线文案和热点清单 | 用户首先感知到的是 baseline 还是长列表 | 当前行为 |
| B4-A | 真实漏判案例收集 | 记录 AI / 人肉原本没察觉、`decay` 抓到且人工认同的案例 | 是否已经形成强案例链 | 价值深度 |
| B5-A | 信任损耗记录 | 记录每次忽略告警的原因 | 噪音、误报、低行动性，还是策略接受 | 价值风险 |

### 5.3 通过标准

局部膨胀价值验证的通过，不等于“能抓受控退化”，而是分两层:

#### 第一层: 机制成立

- `check / diff` 能稳定抓到受控局部膨胀
- 输出能支撑“要不要返工”的决策

#### 第二层: 价值成立

- 至少出现一次“人或 AI 原本会漏掉，但工具抓到了”的真实案例
- 用户没有因为噪音快速失去信任

当前版本最多只能说 **第一层成立，第二层未完成**。

## 6. 推荐执行顺序

1. 跑 `cargo test`，确认实现层没有回归。
2. 跑扫描对象验证:
   - 自测仓库默认热点
   - 自测仓库排噪热点
   - `ripgrep` 默认 / 排噪对比
3. 跑局部膨胀机制验证:
   - 无 baseline 提示
   - `init`
   - clean `check`
   - 受控退化 `check`
   - 受控退化 `diff`
4. 最后补价值深度证据:
   - 真实漏判案例
   - 告警忽略日志

## 7. 2026-05-04 当前结果归纳

### 7.1 扫描对象

当前已验证出的事实:

- 默认扫描会把 `tests/fixtures` 带进自测结果
- 在真实仓库里，`examples` 会稀释热点结果
- 通过 `--exclude` 可以改善结果，但这说明当前默认扫描对象仍然不稳定

当前结论:

- **当前实现能扫描 Rust 源文件**
- **但默认扫描对象还不等于“主维护面源码”**
- **扫描对象问题已被验证暴露，但尚未被产品实现彻底解决**

### 7.2 局部膨胀

当前已验证出的事实:

- `check / diff` 能稳定抓到受控退化
- 输出足够支持 commit 前返工决策
- `hotspots` 有辅助价值，但不是主价值链路

当前结论:

- **局部膨胀机制已经成立**
- **局部膨胀的真实用户价值只完成了部分验证**
- **主要缺口不是再加 metric，而是补真实漏判案例和降噪**

## 8. 下一轮最该补的验证

不是继续加更多“命令跑通”测试，而是补下面 3 类证据:

1. **扫描对象对比验证**
   - 同一仓库在默认扫描、排噪扫描、未来 `prod scope` 下输出差异
2. **真实决策改变验证**
   - 因 `decay check` 改变 commit 决策的真实日志
3. **信任验证**
   - 告警被忽略的原因分布

## 9. 一句话结论

当前验证最重要的结论不是“工具能跑”，而是:

- 扫描对象问题已经被明确暴露
- 局部膨胀主链路已经跑通
- 但产品价值还没有深到可以宣布“已经被完全验证”
