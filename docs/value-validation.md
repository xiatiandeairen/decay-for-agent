# decay 用户需求侧价值验证

> 日期: 2026-05-04
> 目的: 不验证“功能能不能跑”，而验证“用户为什么要用、用了之后是否真的更好、现在的价值到底有多少”。

## 1. 先对齐: 用户不是要“复杂度指标”，而是要“改动裁决”

对当前 v0.1 用户，真实需求不是:

- 我想看一份复杂函数排行榜
- 我想知道这个仓库哪里复杂

真实需求是:

- 我刚让 AI 改完一波代码，准备 commit
- 我不信 AI 对自己代码的自评
- 我也不想花 5 到 15 分钟人肉读 diff 脑算复杂度
- 我现在只想知道一件事: **这次改动里，有没有哪个函数明显变坏了，值不值得当下返工**

所以 `decay` 的价值不在“发现复杂代码”，而在“替用户做一次可重复的 commit 前裁决”。

## 2. 用户原来的替代方案

用户今天不用 `decay`，并不是“什么都做不了”，而是会用下面三种替代方案:

| 替代方案 | 用户为什么会用 | 真正问题 |
|---|---|---|
| 人肉读 `git diff` | 零额外工具成本，马上能做 | 慢，不稳定，依赖状态和经验，容易漏掉局部复杂化 |
| 让 AI 自评 | 快，交互自然 | 有明显 sycophancy bias，同一份改动结论不稳定 |
| 跑 linter / complexity 工具 | 客观，有现成工具 | 看到的是绝对值，不是“这次改动到底让哪里更糟” |

因此 `decay` 只有在下面条件同时成立时才有价值:

1. 比人肉判断更快
2. 比 AI 自评更稳定
3. 比普通 complexity 工具更贴近“本次改动裁决”

少一条，价值都会明显下降。

## 3. 正确的价值验证问题

从用户需求侧，v0.1 真正该验证的不是“扫描成功率”之类工程问题，而是下面 5 个问题:

| 编号 | 问题 | 如果答案是否定，意味着什么 |
|---|---|---|
| D1 | 用户能不能在 commit 前 30 秒内完成一次判断？ | 工具会掉出主工作流，价值很低 |
| D2 | 输出能不能直接支持“现在要不要返工”这个决策？ | 工具只是信息展示，不是决策工具 |
| D3 | 工具能不能抓到人和 AI 都容易漏掉的退化？ | 没有新增价值，只是重复已有判断 |
| D4 | 首次接入成本会不会高到让用户懒得用？ | 再准确也进不了日常习惯 |
| D5 | 告警噪音会不会高到让用户不再相信它？ | 用户一旦失去信任，产品价值迅速归零 |

## 4. 基于当前证据的价值判断

下面判断基于这轮自测、`ripgrep` 验证和现有 PRD/roadmap 目标，不是臆测。

### 4.1 D1: commit 前 30 秒内完成判断

结论: **基本成立**

证据:

- 在自测副本上，`check` 约 `0.08s`
- 在 `ripgrep` 上，`check --exclude examples` 约 `1.43s`
- 用户日常只需要 `check`，不需要每次都读热点列表

用户价值:

- 已经足够进入“改完顺手跑一下”的工作流
- 这点是成立的，否则后面都不用谈

### 4.2 D2: 是否支持“要不要返工”的决策

结论: **成立，但只在 `check / diff` 上成立**

证据:

- 受控退化后，`check` / `diff` 都能精确指出:
  - 哪个函数
  - 哪个 metric 上升
  - 是否跨阈值
- 这类输出能直接支持“这次改动需要拆分 / 回退 / 重构”的动作

限制:

- `hotspots` 只能回答“哪里一直很复杂”
- 它不能回答“这次改动值不值得返工”

用户价值:

- `check / diff` 有强决策价值
- `hotspots` 只有辅助价值

### 4.3 D3: 是否抓到人和 AI 容易漏掉的退化

结论: **目前只得到“部分成立”**

为什么不是“完全成立”:

- 工程上已经证明它能抓到受控退化
- 但“受控退化”还不等于“真实用户本来会漏掉的退化”
- PRD 里真正的核心假设是: 至少出现一次“作者/AI 没察觉，但工具抓到了”的真实瞬间
- 这件事现在文档里还没有形成足够强的真实案例链

这意味着:

- 产品机制是对的
- 但“需求强度”还没被完全打实
- 现在更像“有希望的工具”，还不是“已证明不可替代的工具”

### 4.4 D4: 首次接入成本是否足够低

结论: **对 dogfood 用户成立，对外部用户只部分成立**

证据:

- `init/check/diff/hotspots` 语义已经清楚
- 无 baseline 时有明确引导
- 本地 CLI 工作流简单

限制:

- 需要用户理解 baseline 概念
- 需要用户知道何时该用 `--exclude`
- 当前只支持 Rust
- 还没有 CI/agent 集成，也没有更顺滑的默认项目排除策略

用户价值:

- 对作者本人这种高意愿用户，门槛可接受
- 对普通外部开发者，这个门槛还偏高

### 4.5 D5: 告警噪音会不会伤害信任

结论: **这是当前最大价值风险**

证据:

- `ripgrep` 即使排除 `examples`，首次 baseline 后仍有 `69` 个热点
- 自测仓库会把 `tests/fixtures` 带进来
- `params` metric 在真实项目上的 actionability 偏弱

为什么这是价值问题，不只是“体验问题”:

- 用户不是来读长报告的
- 用户是来得到一个可信裁决
- 如果输出里低价值项太多，用户很快会形成“这工具总在喊复杂，但不值得我动手”的认知
- 一旦形成这种认知，`check` 的高价值信号也会被一起折损

## 5. 当前价值到底有多少

为了避免伪精确，这里不用百分制，改用 4 档判断。

### 5.1 对作者本人 dogfood 用户

结论: **中高价值**

原因:

- 高频场景存在
- 运行足够快
- `check / diff` 已经能直接服务 commit 前裁决
- 比“人肉读 diff”更稳定，比“AI 自评”更可重复

但还没到“高价值 / 不可替代”，因为:

- 真实漏判案例还不够硬
- 噪音控制还不够好
- 热点视图仍容易稀释产品定位

### 5.2 对一般 Rust + AI 编程用户

结论: **中价值以下**

原因:

- 核心需求存在，但证据主要仍来自作者自用
- 默认排除、默认阈值、默认 workflow 还没经过外部用户校准
- 还缺少“我为什么不用 lizard + diff 自己看”的强对比胜出证据

这不是说产品没价值，而是说:

- **价值方向对了**
- **价值强度还没被市场化验证**

## 6. 最关键的结论

### 6.1 已经被验证的

- `decay` 已经证明自己不是“纯技术玩具”
- 在 commit 前复杂度裁决这个点上，它确实提供了独立于 AI 自评的客观判断
- `check / diff` 这条主链路是对的

### 6.2 还没有被验证的

- 用户是否真的会长期养成“每次 AI 改完都跑一下”的习惯
- 工具是否稳定抓到“原本会漏掉，但一旦抓到就值得立即返工”的真实案例
- 噪音是否会在真实长期使用中侵蚀信任

### 6.3 所以现在最准确的产品判断

不是:

- “价值已经被完整验证”

而是:

- “核心需求对得上，主解决方案也对得上”
- “但当前只完成了可用性验证和部分价值验证”
- “距离‘价值被完全证明’还差真实漏判案例和长期信任验证”

## 7. 下一轮应该怎么验证价值

如果目标是验证“用户到底会不会真的持续用”，下一轮不该继续增加工程测试，而该增加下面 3 类证据。

### 7.1 真实决策日志

每次 AI 协作后记录:

- 本来准备直接 commit 还是准备继续改
- `decay check` 报了什么
- 最终有没有因为它而返工
- 如果返工，节省的是未来哪一类成本

这是验证“工具是否改变了决策”的核心证据。

### 7.2 漏判捕获案例

重点收集:

- 人肉 review 没看出来
- AI 自评说没问题
- 但 `decay` 抓到了
- 且事后人工认可“这个告警是对的”

这是验证“不可替代价值”的核心证据。

### 7.3 信任损耗日志

每次忽略告警时记录:

- 为什么忽略
- 是噪音、误报、低优先级，还是知道复杂但故意接受

这样才能区分:

- 产品没价值
- 产品有价值但噪音太大
- 产品有价值但阈值/默认策略没调好

## 8. 最终判断

当前 `decay` 的状态更准确地说是:

- **问题对了**
- **核心切口对了**
- **主链路跑通了**
- **价值已被部分证明，但还没被打穿**

如果只问一句话:

- 对作者本人，现在已经值得继续 dogfood
- 对更广泛用户，现在还不够证明“非用不可”
